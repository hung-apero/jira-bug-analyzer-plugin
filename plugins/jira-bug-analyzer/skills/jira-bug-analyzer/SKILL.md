# Jira Bug Analyzer

Pipeline: choose model → select mode → gather context (board, ticket) → **ask for spec/figma (mandatory) then capture source-of-truth in the background (non-blocking)** → **claim the ticket (In-Progress lock) the moment you start on it** → analyze (use memory if the capture landed, else the dev's provided refs) → ask clarifying questions if needed → short fix plan → only then branch, fix, update Jira.

**Order matters:** analyze first, ask before assuming, plan before coding. **The source-of-truth digest never blocks** — only the mandatory ask for spec/figma links does.

**Scope:** Jira ticket analysis and bug investigation. Does NOT handle ticket creation or sprint management.

## Golden rules — the single source for cross-cutting invariants
The phase/helper files cite these by **tag** (e.g. `[REST]`) instead of restating them. If a rule lives here, it is NOT repeated elsewhere.

- **`[REST]` MCP can't comment / worklog / transition / download attachments.** It returns `"Issue updated successfully"` and silently drops them. Always use REST for those and **verify the result** (re-read; never claim success from a single `2xx`). How-to: `references/helpers/jira-rest-api.md`. *(N/A under `[SHEET]` — a Google-Sheet board makes no Jira calls at all.)*
- **`[SHEET]` A Google-Sheet board replaces the Jira board pull; the sheet is READ-ONLY; status is still saved (to the ledger).** Active when the invocation carries `--google-sheet <URL>` OR the resolved setup has `type: "google-sheet"`. The FULL spec is `references/google-sheet-board.md` — cited by this tag, not restated. The whole delta: **(1)** the board pull is `assets/sheet-board.sh fetch <csvUrl>` (CSV export → header-row detection → EN+VN column auto-detect → rows with attachment URLs harvested from the details cell), and the parsed rows are **cached in `setup.json` local + shared (`[BGMEM]`) and refreshed in the background every turn** — not a live pull; **(2)** there are **no Jira writes** — `[CLAIM]` becomes a `record_bug_status.md` ledger row (the lock), and Phase-6 "finalize Jira" becomes a ledger update (`done` + PR link) — **the sheet is never written back**; **(3)** everything else (Phase 2 spec/figma, Phase 3 root-cause+plan, Phase 4 worktree+build, Phase 5 verify, `[REPRO]` `[CLEANFIX]` `[TOKEN]` `[MEDIA]` `[VN]` `[OPTIONS]` `[1COMMIT]` `[AUTO]`) is **identical**. Bug key = `<PROJECT>-SHEET-<N°>`. Does NOT use the Jira MCP — ignore a `jira-mcp`/`jira-creds` ENV block.
- **`[VN]` Vietnamese for everything dev-facing — always CÓ DẤU (full diacritics), dễ hiểu.** Root-cause/fix-plan analysis, every `AskUserQuestion`, the user-verify checklist, and **every comment posted to Jira** (Fix-9 non-code comment, the auto/defer comment, the unreproducible ask-reporter comment, the `[AUTO-FIX EVAL]` note). **NEVER write no-diacritic / telex Vietnamese** (`"khong the reproduce"` is a BUG — write `"không thể tái hiện"`). **Comments on Jira must read cleanly for a non-technical reporter/QA** — plain, complete sentences, no code/stack/paths/`file:line` in the Jira comment body (those go to the dev in chat). Keep technical tokens **verbatim** (code identifiers, `file:line`, class/func names, RC/config keys, log snippets, enum/flag values, the `fix(scope):` prefix) — translate only the prose. **PR title/description and the memory repo (KB, status, scorecard) stay English** (grep/tooling consistency).
- **`[OPTIONS]` Always offer selectable options, never a bare chat prompt.** Every decision asked of the dev (model, mode, status, assignee, verify verdict, code-review verdict, commit, PR, board pick, transition) is `AskUserQuestion` with concrete choices + a short hint each. Free-text only as the "Other" fallback or for genuinely free content (a spec/figma link, a change description).
- **`[CLAIM]` Claim before analyzing.** The moment a ticket is picked for fixing — *before* fetch/analyze/plan — atomically transition it → **In Progress** AND assign it to the **dev account = the Jira token owner (`/myself`)**, not the Claude session account. This removes it from the pool so no other worker/turn duplicates it. Claim fails → stop, don't analyze (someone else holds it). Already In Progress → skip transition, still set assignee. **Under `[SHEET]` there is no Jira transition — the claim IS the `record_bug_status.md` ledger row (`status: analyzing`, keyed `<PROJECT>-SHEET-<N°>`); a matching row owned by another dev is the lock.**
- **`[VERIFY]` Never PR before verification passes — and verification is its own phase (Phase 5).** A fixed ticket goes fix+build (Phase 4 — fix by the plan, then build the app via `./gradlew assembleAppDevDebug`) → **Phase 5** (`references/phase5-verify.md`), in order: **review the diff on the GitHub webview** — **Gate 1: ACTUALLY open the diff in a browser/webview (`gh pr diff --web` / OS `open`/`xdg-open`/`start` / `mcp__claude-in-chrome__navigate`) AND ask `OK`/`Needs change` as its own standalone prompt BEFORE any install. Pasting a compare/PR LINK into chat does NOT satisfy this and is a BUG; never fold it into the user-verify card.** (needs change → back to Phase 4 for impl scope / Phase 3 for plan scope) → **install the Phase-4 APK** (reuse, not rebuild) + **user-verify** → **adb self-verify ONLY on `--auto`** (dispatched as a subagent per `[TOKEN]` — `android-ui-verify` for UI-scoped / `android-self-verify` for behavioral; `am start` / `input` / `uiautomator dump` / `exec-out screencap` / `shell screenrecord` — **NOT mobile-mcp**) — all passing, *then* the `[1COMMIT]` + PR + Jira-Resolved (background) in **Phase 6** (`references/phase6-commit-pr.md`). Never skip user-verify (only the human may waive — **or `--auto`, where adb self-verify IS the gate per `[AUTO]` / `auto-run-test.md`**). **Under `--auto` the self-verify MUST save evidence whose medium matches the bug — a screenshot for a static/visual bug, a screen-recording video for a dynamic/flow/animation/crash bug (both when in doubt) — into `.jira-bug/evidence/<TICKET>/`; a `pass` with no saved evidence file is invalid, and that evidence is carried into the PR body + **EMBEDDED inline in the Jira resolve comment** — attach the file first, then post the comment that references it (`!name.png!` / `[^name.mp4]`), and re-read to confirm it rendered; an attachment the comment never references is invisible to the reporter and counts as NO evidence (Phase 6).** Never fix → PR/Resolved directly.
- **`[1COMMIT]` One commit per ticket; never squash a multi turn.** Single → one commit. Multi/team → exactly one commit per resolved ticket on the shared/round branch; merge via merge-commit/rebase, **never** `--squash`. Conventional + scoped + ticket key: `fix(scope): … (TICKET)`.
- **`[REPRO]` Bugs are diagnosed to the maintainable bug-analysis ruleset — `references/helpers/bug-analysis-rules.md` (`A1`–`A14`).** The analyzer applies it at **Phase 3 Fix-5.5**, before root-causing: **(1) reality-check** — is this a real code bug or a non-code cause (config/backend/stale-build/works-as-designed/duplicate)? cite evidence, else Fix-9/defer (A1–A3); **(2) reproduce on-device FIRST** — acquire the device lock, follow the repro steps, observe via human-mcp, save the repro evidence (reused by Phase 5), trace trigger→symptom to a `file:line` (A4–A6); **(3) device policy** — **`--auto` HARD-REQUIRES a device** (no device / can't reproduce → **defer**, never auto-fix an unproven bug), **interactive degrades** to static repro + a `not device-verified` flag (A7); **(4) can't-reproduce ladder** — drive candidate trigger-cases; none reproduce → **defer as unreproducible + ask the reporter for steps/video/build**, never guess-fix (A8–A10); **(5) rigor** — one evidence-backed root cause, the repro steps become the Phase-5 verify script, no Phase-4 handoff without a confirmed/strongly-evidenced trigger (A11–A14). The confirmed repro keys the `deterministic_repro` scorecard points. **This file is the single place to maintain how bugs are analyzed.**
- **`[CLEANFIX]` Fixes are written to the maintainable clean-code ruleset — `references/helpers/fix-code-rules.md` (`R1`–`R22`).** The fixer reads + follows it at **Phase 4 Fix-11**: a bug fix is **surgical** (smallest diff for the root cause, no unrelated refactor — R1–R4), fixes the **cause not the symptom** (R5–R7), **matches surrounding style with no new comments** (R8–R10), respects **Clean-Architecture/MVI layers + the design system + TeraKit primitives** (R11–R13), introduces **no regression** (R14–R17), and stays **hygienic** (R18–R22). The **Phase-5 review gate checks the diff against these rules** (`code-reviewer` under `--auto`; the dev's webview review interactively) — a material breach bounces the fix back to Fix-11. **This file is the single place to maintain fix-writing standards** — edit it and every project the skill fixes picks it up (a target project's own `docs/code-standards.md` is additive; stricter wins).
- **`[TOKEN]` Read-heavy drives run in a SUBAGENT; the main loop consumes only their text result — never raw media/XML/logs.** The Phase-5 `--auto` self-verify (`android-ui-verify` / `android-self-verify`) is **dispatched as a subagent** (Task tool): it Reads the screencap PNGs, Figma reference images, `uiautomator` XML, `logcat` tails, `eyes_compare` output, and the 5× retries inside ITS context, and returns **only the structured verdict block + the SAVED evidence file PATHS** (`.jira-bug/evidence/<TICKET>/…`). The main loop **never Reads a screenshot/XML/log** — this is the single biggest token sink and verify was the last drive still running inline (Phase-3 analyzers and Phase-6 finalize already isolate this way). Inside any device drive, also cap the noise: Read only the **decisive** screenshot(s), `grep` the one `uiautomator` node's `bounds` (never `cat` the whole XML), `logcat -d -t 100`. Evidence stays durable (saved to disk + PR/Jira) without costing main-loop tokens. **The same rule covers Jira-attachment media analysis (Fix-3–Fix-6): multi mode already fans it out to background analyzers, and SINGLE mode dispatches ONE analyzer subagent too — so attachment images/video are Read in the subagent, which returns only the Fix-6 block text. The main loop never Reads attachment media.** **`[MEDIA]` Analyze images/videos via human-mcp FIRST (`references/helpers/media-analysis.md`): `eyes_analyze`/`eyes_compare` take a file PATH and return TEXT, so the pixels/frames never enter context at all — no per-image token cost, nothing to shrink. Keys are a list in `.claude/.env` (`HUMAN_MCP_GEMINI_KEYS`) that human-mcp rotates internally; the skill caches an "exhausted-today" flag (`.jira-bug/human-mcp-state.json`, daily reset) and only then falls back to Claude-native vision — and ONLY that fallback path uses the shrink recipe (`references/helpers/media-preprocessing.md`). Evidence files are still SAVED full-res to disk for PR/Jira regardless of path.** **`[RAG]` Read the source-of-truth BY SECTION, not whole: Phase-2 digests (`spec.md`/`figma.md`) are structured with a TOC + `## <anchor-slug>` sections and a `sections` index in `metadata.json`; an analyzer matches its ticket's screen/feature to one anchor and reads ONLY that section — never the full spec (`references/phase2-source-of-truth.md` + the Phase-3 Context-source rule).**
- **`[BGMEM]` All remote-memory I/O runs in a BACKGROUND agent — never block the main loop on a memory pull/push.** Every read from or write to the shared memory repo goes through a subagent dispatched with `run_in_background: true` (the `memory-keeper` agent for `status`/`project-setup`/`kb-upsert`/`kb-feedback`, the `source-of-truth` capture, and any raw `memory-sync.sh pull`/`push` — incl. the `record_bug_status.md` ledger writes — the sole per-bug record — the first-run/`baseBranch`/setup pushes, and the dedup-ledger pull). The interactive flow dispatches the background agent and **moves on immediately**; it never awaits the git network round-trip. **"Background" means non-blocking, NOT deferred to a later turn:** a write a phase marks *mandatory this turn* (first-run `setup.json`, `baseBranch`, the analyzer's `analyzed` ledger row) is still **dispatched in that same turn** — it just runs as a background agent instead of blocking, and its completion notification is **consumed lazily** at the next gate that actually needs the result (dedup/claim). The ONLY memory step allowed inline is the Init warm-cache TTL check (`memory-sync.sh pull 600`), which is a local timestamp comparison, not network I/O; an actual network pull/refresh folds into the existing background `REFRESH` subagent. If a background memory push fails, **surface it** (un-pushed memory = dedup silently off / config not inherited) — but still never hold the main loop waiting on it. Memory-write ordering rules elsewhere (re-push the ledger row on each transition, one-file-per-commit) are unchanged — they now happen *inside* the background agent.
- **`[AUTO]` `--auto` is the fully-autonomous loop — confident work proceeds unattended, everything uncertain is parked + reported, the loop never stalls on a human.** **[NO INLINE ASKS — TRY-AND-DECIDE] Under `--auto` the ONLY permitted human stops are the gates this skill EXPLICITLY marks `[AUTO-GATE]` (defined just below). For EVERY other situation — INCLUDING ones no rubric anticipated (a red/broken base branch, a missing/flaky dep, a small out-of-scope prerequisite edit needed just to build, an ambiguous-but-decidable choice, an unexpected tooling/merge state) — the agent MUST try-and-decide autonomously: pick the safe, reversible default, act ONLY inside the isolated fix branch/worktree (never force-push or commit to a shared ref), proceed, and RECORD the decision + rationale in the end-of-run report. An inline `AskUserQuestion` for anything that is NOT an `[AUTO-GATE]` is a BUG — keep the loop moving.** The `[AUTO-GATE]` human stops are exactly: **(1) PR merge** — `--auto` opens/updates PRs but NEVER merges; a human merges on GitHub; **(2)** any stop the dev has explicitly defined for this run. Everything else, including low-confidence / clarify-gap / blocked tickets, is **auto-deferred + reported, NOT asked.** The dev typing `--auto` is the documented **waiver of `[VERIFY]`'s user-verify and `[OPTIONS]`'s always-ask** for confident work. The three *evaluative* decisions each have their own rubric file (cited at their gate): **pick** → `references/helpers/auto-ticket-pick.md` (Phase 1) · **fix vs defer** → `references/helpers/auto-plan-pick.md` (Phase 3, consumes `confidence-rubric.md`) · **pass/fail/blocked** → `references/helpers/auto-run-test.md` (Phase 5: `code-reviewer` diff gate + adb self-verify). The *fixed defaults* (non-evaluative) live HERE:
  - **Deferred tickets go back to `Request`, then the loop CONTINUES — a defer is NEVER a stop condition.** When a ticket defers (score <80 / clarify-gap / mid-fix wrong-plan, per `references/helpers/auto-plan-pick.md`), transition it → **`Request`** (out of `In Progress`, back to the open triage pool), comment the reason + plan summary, keep its `analyzed-deferred` ledger row + report it, and **move straight to the next ticket/batch.** Anti-re-pick is NOT by status (it stays in the pull pool) but by **dedup**: `auto-ticket-pick.md` skips any ticket with an `analyzed-deferred` ledger row — EXCEPT one reopened with new info (comment/attachment after the defer date), which is correctly re-analyzed. (Full mechanics: `auto-plan-pick.md` → "Khi defer".)
  - **A fixable infra blocker is NOT a stop condition either** — e.g. a red/broken base branch (`origin/develop` won't compile): fold the minimal unbreak into the current fix branch (or branch subsequent fixes off the batch branch that already carries it), note it in the report, and keep looping. Only an empty re-pull or a dev-defined `[AUTO-GATE]` stops the loop.
  - **Phase 6 (no prompts, RUNS IN A BACKGROUND AGENT):** the whole of Phase 6 — commit · create/update the batch PR · worklog `30m` · attach PR link + transition Jira → Resolved — is **dispatched as a background agent (`run_in_background: true`)** so the main loop NEVER blocks on the commit/PR/Jira round-trips. Defaults: auto-commit · auto-create PR · worklog `30m` · **Discord OFF unless `--discord` was passed** (a plain auto batch never posts; with `--discord` the background Phase-6 agent enqueues the opened PR to Discord via `pr-discord-review-request` — best-effort, a failed/absent `PR_DISCORD_CHANNEL_URL` is logged to the report and skipped, never blocks the loop) · **stamps the auto-fix quality flag** (Jira label `auto-fixed` + `[AUTO-FIX EVAL]` comment, reopen-outcome `pending`) so `--manager` Job D can later record whether the fix held or got reopened — the metric for evaluating auto-fix quality. Auto opens PRs but **NEVER merges** — human review stays at the GitHub PR before merge (the safety net). **PR review-comment handling / worktree cleanup / KB-backfill are NOT armed inline — they are handled by `--manager` (`references/phase-manager-mode.md`); run it after an `--auto` batch to surface review comments, clean up worktrees, and backfill the KB.** The `--auto` looper still writes `.jira-bug/pr-watch.json` per opened PR so manager mode has entries to reconcile.
  - **Loop until the board is empty (the `--auto` looper):** after dispatching Phase 6 in the background for the drained batch, **immediately loop back to Phase 1 (re-pull)** and auto-pick the next batch — repeat Phase 1→6 unattended **until a re-pull returns ZERO tickets**, then write the end-of-run report and STOP. The loop never waits on a human and never awaits the background Phase-6 agents (their completion notifications are consumed lazily; a failed background Phase 6 is surfaced in the report, never blocks the loop).
  - **Retry budget:** max **2** re-fix attempts/ticket across Gate-1-critical + Gate-2-fail combined (`auto-run-test.md`); exhausted → defer. (Distinct from the **5×** verify-observation retry in `auto-run-test.md`.)
  - **End-of-run report** → write `.jira-bug/auto-report-<date>.md` AND surface in chat: ✅ PRs opened (ticket→PR link) · ⏸️ Deferred (ticket, score, reason: `<80` / `clarify-gap` / `no-spec` / `mid-fix wrong-plan`) · 🚫 Blocked (ticket, reason: `claim-fail` / `adb-blocked` / `retries-exhausted` / `feature-gated` — hidden by a remote-config/feature flag that couldn't be force-enabled; see Phase-5 `[FEATURE-GATE]`). Reuses `.jira-bug/batch-progress.md` (statuses extended: `deferred`, `blocked-auto`).
  - **Scope:** multi `--auto` = full loop; single `--auto` = the same gate decisions for the one ticket (pick is N/A). `--auto` absent → every gate is interactive as before.

## Mode dispatch — pick the ONE init file before anything else

Mode is resolved from the invocation args FIRST (this is the only thing that decides which init file to read). Resolve in this order, then read exactly that file and follow it:

0. **`--google-sheet <URL>` present (a board-SOURCE modifier, not a mode)** → the board is a link-shared Google Sheet, not a Jira board (Golden Rule **`[SHEET]`**; full spec `references/google-sheet-board.md`). It **composes with multi/team, never single** (a sheet IS a board): strip the `--google-sheet <URL>` token, resolve the mode by the rules below (a lone project key or no token + `--google-sheet` ⇒ **multi-without-team**; `--team` ⇒ team), and pass `--google-sheet <URL>` through to the init script. At the pull step follow the sheet branch (`references/phase1-init-multi-mode-without-team.md` → "Pull & list" → the `SETUP_TYPE=google-sheet` case). A run whose resolved setup already carries `type: "google-sheet"` behaves the same even without the flag. Still needs a `--project <KEY>` for memory namespacing — ask if absent. Does NOT use the Jira MCP; ignore a `jira-mcp`/`jira-creds` ENV block on this path.
1. **`--team` present** → **multi WITH team** → read **`references/phase1-init-multi-mode-with-team.md`** (this session becomes MainCharacter; pair with `team/team.md`). `--team` overrides worker modes; `--devs N` sets the Dev count. Without `--team`, the skill is the solo worker — never launches a team.
2. **`--resume` present** → a modifier, not a mode (Init-5 reconcile). It accompanies whichever mode the rest of the args select; resolve the mode by the other rules, read that file, do its `--resume` reconcile. Off by default — a plain invocation never resumes.
3. **`--manager` present** → **manager mode** (the 3rd mode — handles manager-level work: PR review comments, worktree cleanup, KB-backfill — NOT the fix pipeline) → read **`references/phase-manager-mode.md`**. Requires a `<PROJECT-KEY>` (the board) — if absent, ASK. **`--manager` WINS over token-shape inference** (a bare project key alongside it is the board, not a multi-pull). Reuses init plumbing but skips board pull / pick / analyze / fix. **Manager mode ALWAYS self-arms a recurring `/loop` (default every 10 min) on entry** — see that file's "Self-arm the `/loop`" section; it is never a bare one-shot.
4. **`--mode single <KEY>`** *(or a bare **ticket-key** token `[A-Z][A-Z0-9]+-\d+`, e.g. `AIP686-179` — note the `-<number>`)* → **single** → read **`references/phase1-init-single-mode.md`**. `--mode single` requires a key — if absent, ask for it. The board key is the ticket's prefix (`AIP686-179` → `AIP686`), passed to `init-single.sh --project AIP686`.
5. **A bare *project-key* token `[A-Z][A-Z0-9]+` with NO `-<number>`** *(e.g. `AIP686`)* → **multi without team**, and **that token IS the board** → read **`references/phase1-init-multi-mode-without-team.md`** and run `init-multi.sh "<root>" --project <KEY>`. Emit one line (*"project key `AIP686` → multi mode, board=AIP686"*). (This is the common `/jira-bug-analyzer AIP686` invocation.)
6. **`--mode multi`** *(also `pull`/`batch` aliases — emit one line "`pull`/`batch` renamed to `multi`")*, **or no args at all + no `--team`** *(the default)* → **multi without team** → read **`references/phase1-init-multi-mode-without-team.md`**. **No project key was given → ASK for it immediately (`AskUserQuestion`) as your first action — before running init. Do NOT infer the board from the working directory / repo / `CLAUDE.md`, and do NOT silently adopt a cached board.** Offer the local cached board (if any) as the recommended one-tap option. See that file's **ASK-FIRST gate**.
7. Any other `--mode` value → reject with the usage line.

> **Disambiguation:** a token's shape decides single vs multi — `KEY-123` (has `-<digits>`) is a **ticket** → single; `KEY` (no trailing `-digits`) is a **project key** → multi board. When in doubt (e.g. an odd token), `[OPTIONS]`-ask whether it's a ticket or a board. **`--manager` overrides shape inference** — a bare `KEY` next to it is just the board for the manager reconcile, never a multi-pull.
> **Phase resolution (`@N`) — which `<PHASE>/doc/` + `<PHASE>/session/` + `phaseX/` branch prefix this run uses.** A bare `@N` token in the args (e.g. `@3`) sets **`phase = phaseN`** explicitly — this WINS over everything. If no `@N` is given, the phase is **derived from the spec's own title** at the Source-of-truth capture (Phase 2): the `source-of-truth` subagent reads the Confluence page title(s)/parent (e.g. a "Phase 3 / Sprint 3 / Milestone 3" marker) and uses that phase. Only if neither an `@N` arg nor a phase-bearing spec title exists → fall back to `phase1`. **Never silently default to `phase1` when the spec title says otherwise** — a wrong phase folder splits memory and dedup. If `@N` and the spec title disagree, honor `@N` but surface the mismatch. Strip the `@N` token before mode resolution — it is orthogonal to single/multi/team.
> **Usage:** `/jira-bug-analyzer <PROJECT_KEY>` (multi on that board) · `/jira-bug-analyzer <PROJECT_KEY> @N` (multi, phase N — e.g. `AIP686 @3`) · `/jira-bug-analyzer <TICKET-KEY> [@N]` (single) · `/jira-bug-analyzer --google-sheet <SHEET_URL> --project <KEY> [@N] [--auto]` (**multi on a Google Sheet board** — `[SHEET]`) · `/jira-bug-analyzer --team [<PROJECT_KEY>] [@N]` (team) · `/jira-bug-analyzer --manager <PROJECT_KEY> [@N] [--auto]` (manager mode — PR review comments + worktree cleanup + KB-backfill; **always self-arms a recurring `/loop`, default every 10 min**) · `--discord` (opt-in Discord PR-review posting, any mode incl. `--auto`) · `--resume` (reconcile). Always pass the resolved project key to the init script as `--project <KEY>` so it can resolve setup local→remote.
> **Flags:** `--auto` (**fully-autonomous loop — see Golden Rule `[AUTO]`**: auto-pick (`auto-ticket-pick.md`) → auto fix-vs-defer by confidence (`auto-plan-pick.md`) → `code-reviewer` + the verifier subagent (`android-ui-verify` UI / `android-self-verify` behavioral) as the gate, 5× verify retries (`auto-run-test.md`) → **auto commit+PR in a background Phase-6 agent (Discord OFF unless `--discord`)** → **loop back to Phase 1 and repeat until the board is empty** → end-of-run report; `<80`/clarify-gap/blocked tickets are parked + reported, never guessed; PRs opened but never merged; **the inline PR watcher is NOT armed — review-comment handling + worktree cleanup + KB-backfill are handled by `--manager`, run it after the batch**. Default OFF = every gate interactive). When used with `--manager`, `--auto` governs Job A only (auto-apply unambiguous review fixes; Job C KB-backfill is unattended regardless). `--manager` *(the 3rd mode — manager mode: PR review comments + worktree cleanup + KB-backfill; **always self-arms a recurring `/loop`, default every 10 min, on entry**; see `references/phase-manager-mode.md`)*. `--pr-window <N>m` (**removed** — review is now per-ticket & streamed the instant each fixer is green, no windowed round; warn+ignore if passed). `--discord` (**opt-in Discord PR-review posting, any mode**: enqueue each opened PR to the *Apero Mobile Developer / review-pr* channel via `pr-discord-review-request`, Phase-6 Step 3. In **interactive** modes it turns the always-asked *"Post review request to Discord?"* prompt into auto-yes — skip the ask, always enqueue. ## Multi mode — the turn loop

The multi-mode orchestration loop, **spanning Phases 1–6**. The per-phase detail lives in each phase file; this is the loop that drives them. (Single mode = one bug, no fan-out — run the single workflow inline: Fix-1–Fix-8 in `references/phase3-analyze.md` → Fix-9–Fix-12 in `references/phase4-fix-and-build.md` → Phase 5 → Phase 6.)

**pull → list → dev picks indices → fix the picked set by their approved plans + build each to verify it compiles (fan-out) → as EACH ticket turns green, immediately show its diff + auto-open the browser → on OK merge it into the batch branch first (no waiting for others) → user-verify → **Phase 6 (background agent) opens/updates the batch PR** → drain → re-pull next turn → repeat until the board is empty or the dev picks nothing.** Plans approved BEFORE any code (same as single). Analysis, fixing, AND review/merge all stream per ticket and overlap; no windowed round — each ticket ships the moment it's ready. **Under `--auto` the whole thing is a self-driving loop: Phase 1→6 unattended (Phase 6 dispatched in the background), looping until a re-pull returns zero tickets.**

**Never re-pull during an active fix** (a branch checked out / device lock held). Re-pull only **between turns**, after the picked set is fully drained.

### The turn loop
1. **Pull** (Pull & list by category block, `references/phase1-init-multi-mode-without-team.md`) → numbered list grouped by natural Category. Empty → STOP (report "no <status set> bugs left on <board>", no background polling).
2. **Dev picks indices** `[OPTIONS]`/free-text (e.g. `1,3,4`). Picks nothing → STOP. **Cap ≤4/category per turn** — overflow → top 4 by priority, explicitly note the deferred (re-surface next turn). **A category selection (group header/name) = pick all bugs in that category (≤4 by priority) and PROCEED — never ask a confirmation follow-up; never re-confirm any pick the dev already made.** Go straight to claim/analyze.
3. **Source-of-truth capture (background, non-blocking — Phase 2,** `references/phase2-source-of-truth.md`**)** — ensure the spec/figma/other refs were provided (mandatory ask) and the `source-of-truth` capture is **launched in the background**. Do **not** WAIT for the digest. Analyzers (step 6) resolve their SOT context via the Context-source rule (memory if ready, else the provided refs). Skip re-launch on later turns of the same phase unless flagged stale or an external link is still pending (re-try the direct read).
4. **Claim all** `[CLAIM]` every picked ticket up front (before any agent) — removes them from the pool. A claim that fails → drop that ticket.
5. Write `.jira-bug/batch-progress.md` listing every claimed ticket `pending`.
6. **Analyze fan-out (background dispatch → FIFO completion queue → single interactive floor)** — see `references/phase3-analyze.md` → "Multi mode — analyze fan-out". **Dispatch every picked ticket's analyzer as a background agent (`run_in_background: true`) in one message**; present plans **one at a time in completion order**, holding the floor for one disposition per ticket then releasing (deferred completions enqueue, never interrupt). `plan-ready` + Approve → mark `approved` → **launch its fixer in the background now** (step 8) → release the floor; `Needs changes` → re-dispatch revised analyzer in background → release the floor (re-enters the queue when it lands).
7. *(handled inside Phase 3 — per-analyzer returns: `plan-ready` / `duplicate` / `comment-needed` / `blocked`.)*
8. **Fixer per approved ticket** (background agent, parallel — Multi fan-out block; Phase 4): launched on Approve (does not block the floor), hand it the approved plan verbatim, mark `fixing`. It fixes **by the plan**, then runs `./gradlew assembleAppDevDebug` to **verify the app compiles**. Green build → returns `fixed-in-worktree` → mark `ready` + make its `[1COMMIT]`; red build → back to fixing; `blocked` (incl. wrong-plan) → loops back to Phase 3 step 2 for a re-approved plan.
9. **Per-ticket review & batch-merge — streamed, NO window wait** (Per-ticket review & batch-merge block): the **instant** a ticket turns `ready`, run **Phase 5** (`references/phase5-verify.md`) for THAT ticket — **show its diff + auto-open the dev's browser** on it, ask `OK`/`Needs change`; **`OK` → merge it into the batch branch `phaseX/fixbug/batch-<YYMMDD>` FIRST, immediately (do NOT wait for other tickets)** → install its APK + user-verify `[VERIFY]` (adb self-verify only on `--auto`) → **Phase 6** (`references/phase6-commit-pr.md`) — **dispatched as a BACKGROUND agent** (`run_in_background: true`): the whole commit + open/update the batch PR (carrying that ticket's `[1COMMIT]`) + Jira finalize runs off the main loop, which never blocks on it. `Needs change` → loop that ONE ticket back to Phase 4 (impl) / Phase 3 (plan); other tickets keep streaming independently.
10. **Turn cleanup:** worktree cleanup + PR review-comment observation/fixing + KB-backfill are **handled by `--manager`** (`references/phase-manager-mode.md`) — run it post-batch; they are NOT armed inline in this loop. Delete `.jira-bug/batch-progress.md` once every ticket in the batch is drained (Resolved / deferred / blocked).
11. **Re-pull (next turn / `--auto` loop):** drained → back to step 1 with the same board/status/assignee + spec/figma (no re-ask). New + previously-deferred bugs resurface. **Fire the re-pull from the in-context `PULLQUERY_JQL` immediately** (already emitted in this session) — do NOT wait on `init-multi.sh`; run it in the **background** (`run_in_background: true`) in the same message for env/memory revalidation, and **render the list the instant `jira_search` returns**. Reconcile only if the background init reports a changed `BOARD`/`PULLQUERY_JQL` (re-fire) or `ENV=block:<dep>` (handle the block). (warm fast-path → "Re-pull turns".) **Under `--auto`: this re-pull is automatic (no human pick) — keep looping Phase 1→6 (Phase 6 in the background) until a re-pull returns ZERO tickets, then write the end-of-run report and STOP.**

> **Single-key shortcut:** dev picks exactly one index → run the single workflow inline (Fix-1–Fix-8 analyze in `references/phase3-analyze.md` → Fix-9–Fix-12 fix in `references/phase4-fix-and-build.md` → Phase 5 verify → Phase 6 commit+PR) instead of the fan-out, then re-pull.

**Boundaries:** analyzers read-only (no worktree/edits/Jira — Phase 3); fixers edit only in their own worktree, only after approval; **Jira writes happen in this session / the Phase-6 background agent after the relevant gate** (no analyzer/fixer posts to Jira). **Stop conditions (interactive):** empty pull · dev picks nothing · dev interrupts. **Stop condition under `--auto`: the ONLY stop is an empty pull (zero tickets)** — it loops Phase 1→6 unattended, Phase 6 dispatched as a background agent, re-pulling each turn until the board is empty, then writes the end-of-run report. Loop runs only while the session is open.

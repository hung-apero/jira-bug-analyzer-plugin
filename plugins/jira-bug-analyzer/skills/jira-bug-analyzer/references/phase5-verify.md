# Phase 5 ‚Äî Verify (review diff on GitHub ‚Üí build/install ‚Üí user-verify ‚Üí PR; `--auto` self-verify via the `android-self-verify` skill)

> The flow's fifth phase: **Intake (phase1) ‚Üí Source-of-truth (phase2) ‚Üí Analyze (phase3) ‚Üí Fix + build (phase4) ‚Üí Verify (THIS phase) ‚Üí Commit & PR (phase6).** A fix arrives here **already compiled AND built into an APK in its worktree** (Phase 4 Fix-12 ran `./gradlew assembleAppDevDebug`). This phase: (1) shows the **code change in the GitHub webview** for the dev to review, (2) on no-change, **installs the APK Phase 4 already built** and gets the human verify, (3) on user-verify pass, hands to **Phase 6** to open the PR + attach it to Jira + move the ticket to Resolved. **Self-verify: for a UI-scoped ticket the design-anchored `android-ui-verify` (device ‚ü∑ Figma 4-axis diff + blast-radius sweep) is MANDATORY on the verify gate ‚Äî interactive AND `--auto` ‚Äî and may NOT be reduced to a "looks present / compiled" smoke-check; the behavioral `android-self-verify` runs ONLY under `--auto`. Both drive via screenshot-observed `adb` ‚Äî not mobile-mcp.** Invariants are SKILL.md Golden Rules (`[VERIFY]` `[1COMMIT]` `[VN]` `[OPTIONS]` `[REST]`). Shared blocks (PR round, watcher, device-lock, Base branch) live in `references/run-blocks.md` / `references/helpers/`.

## Entry contract
Arrives from Phase 4 with: the fix in its worktree, **compiled and assembled into an APK** (the Phase-4 build). `<BASE>` = the resolved `setup.json.baseBranch` (e.g. `develop` ‚Äî never a hardcoded `main`; Base branch block in `references/run-blocks.md`). Carry forward the Phase-3 analysis (root cause, scorecard, plan, files) ‚Äî it feeds the user-verify card and the PR description.

## The gates (in order)

### Gate 1 ‚Äî Review the code change (FIRST, before any device work) ‚Äî per ticket, IMMEDIATELY, auto-open the browser

> **[HARD SEQUENCING BARRIER ‚Äî Gate 2 MUST NOT begin until Gate 1 is fully done.]** Gate 1 is satisfied **only** when BOTH have happened: (1) the diff was **actually opened in a browser/webview** (`gh pr diff --web`, the OS opener `open`/`xdg-open`/`start`, or a `mcp__claude-in-chrome__navigate` to the compare/PR URL), AND (2) the dev answered the standalone `OK` / `Needs change` question. **Pasting a GitHub compare/PR LINK into chat is NOT Gate 1 ‚Äî it does not open the webview and does not ask the gate; doing only that and proceeding to install/user-verify is a BUG (the exact miss this barrier prevents).** Do **not** fold the diff review into the Gate-2 user-verify card: Gate 1 (webview diff + `OK`/`Needs change`) and Gate 2 (install APK + user-verify) are **two separate prompts in order**, never merged into one. Before you run ANY Gate-2 step (device lock, install, launch), self-check: *did I open the browser on the diff AND get an explicit `OK` this run?* If not ‚Üí go back and do Gate 1.

**Trigger the instant a ticket's fixer returns green ‚Äî do NOT wait for other tickets, do NOT batch a window.** **AUTO-FIRE ‚Äî never ask permission to enter Gate 1.** The moment the fixer returns green, push the WIP + open the browser **unconditionally**. Do **NOT** prompt the dev with a "proceed to review? / what next? / shall I open the diff?" / "Review diff vs Install APK vs Pause" style question ‚Äî that pre-gate is a BUG. The fixer being green IS the trigger; opening the diff is automatic. The **ONLY** question Gate 1 ever asks is `OK` / `Needs change`, and it is asked **AFTER** the diff page is already open (so the dev decides while looking at it). Push a render-only WIP commit so GitHub has something to render, then **automatically OPEN the dev's browser on the diff** (don't just print the link):
```bash
git -C <wt> add -A && git -C <wt> commit -m "wip: <TICKET> review" && git -C <wt> push -u origin <branch>
# PR/draft exists ‚Üí gh pr diff <pr> --web        # opens the browser
# else open the compare view via the OS opener:
#   Windows: start "" "https://github.com/<owner>/<repo>/compare/<BASE>...<branch>?expand=1"
#   mac/linux: open / xdg-open / python -m webbrowser "<compare-url>"
```

This card is pulled straight from the Phase-3 analysis (`Title`/`Description`/`Resource link`/`Root cause`/`Fix plan`/`File`) carried into this phase ‚Äî never make the dev open Jira separately to recall what the bug was. Then ask `[OPTIONS]` (`OK` / `Needs change`) ‚Äî **this is the FIRST and ONLY prompt of Gate 1, asked only after the browser is already open on the diff AND the bug-detail card is shown.** Never precede it with a "want to review?" / "what next?" question. **Decide the loop-back by the SCOPE of the change:**
- **Needs change ‚Äî implementation scope** (wrong/incomplete code, missing edge case, wrong file, style, **or any clean-code-rule violation ‚Äî `references/helpers/fix-code-rules.md` `R1`‚Äì`R22`**: non-surgical/unrelated refactor, symptom band-aid, layer/design-system breach, regression risk, leftover debug/dead code) ‚Üí back to **Phase 4 Fix-11**: re-fix ‚Üí re-build (`assembleAppDevDebug`) ‚Üí `git -C <wt> commit --amend` the WIP + `push --force-with-lease` ‚Üí **re-enter Gate 1** ‚Äî that ONE ticket only; other tickets keep streaming.
- **Needs change ‚Äî root-cause / plan scope** (the diagnosis or the approved approach itself is wrong) ‚Üí back to **Phase 3** (`references/phase3-analyze.md`): re-analyze ‚Üí revise the plan ‚Üí re-approve ‚Üí **Phase 4** (re-fix + re-build) ‚Üí **re-enter Gate 1**.
- **OK ‚Üí merge THIS ticket into the batch branch FIRST, immediately** ‚Äî cherry-pick its `[1COMMIT]` onto the batch's main branch `phaseX/fixbug/batch-<YYMMDD>` (created off `origin/<BASE>` by the first ready ticket; Per-ticket review & batch-merge block in `references/run-blocks.md`). **Never wait for another ticket.** Then ‚Üí Gate 2.

>
> **`[AUTO]` (`--auto`) ‚Äî `code-reviewer` IS the gate here (no human No/Needs-change).** Run `code-reviewer` on the diff per `references/helpers/auto-run-test.md`, **checking it against `references/helpers/fix-code-rules.md` (`R1`‚Äì`R22`)**: no **critical** finding ‚Üí proceed to Gate 2; a **critical** finding ‚Üí back to Phase 4 Fix-11 (counts against the 2√ó retry budget). **A material clean-code-rule breach is critical** ‚Äî non-surgical/unrelated refactor (R2), symptom band-aid over the root cause (R5‚ÄìR6), a Clean-Architecture/design-system violation (R11‚ÄìR13), or a regression risk (R14‚ÄìR17). Cosmetic nits are recorded in the PR body, not blocking.

### Gate 2 ‚Äî Install the Phase-4 APK + user-verify (`--auto` self-verify via the `android-self-verify` skill)
1. **Acquire the device lock via `assets/device-lock.sh`** (`references/helpers/device-lock.md`) ‚Äî **MANDATORY, keyed by the RAW Claude session id** (`$TOKEN` ‚Äî the same string for every phase/ticket/subagent of this session; a per-role/per-ticket variant like `<sid>-analyzer-319` self-collides, is rejected with `BADTOKEN`, and is what broke this lock before): `SERIAL=$(bash <skill-dir>/assets/device-lock.sh free-serial "<session-id>" 1800)` (or `acquire <SERIAL> "<session-id>"` for a specific device). Empty / `BUSY` (exit 3) ‚Üí another session/project owns every device ‚Üí wait or tell the user; **never steal a locked device, never bare-`adb install` to skip the lock.**
2. **Install the APK Phase 4 already built ‚Äî THROUGH the lock:** `bash <skill-dir>/assets/device-lock.sh install <SERIAL> "<session-id>" <apk>` (acquire-then-`install -r -d`, keeps the lock held), then launch (`am start -n <pkg>/<launcher>`). **Do NOT rebuild** ‚Äî reuse the Phase-4 APK; rebuild only if Gate 1's loop changed the code (then re-run `assembleAppDevDebug` in that worktree). A bare `adb -s <SERIAL> install` here is a BUG (it lets a concurrent session clobber your build on the device).

> **`[FEATURE-GATE]` The feature/screen under test is hidden by a remote-config / feature-flag gate ‚Üí flip the gate to a TESTING value (test-only, uncommitted) and re-verify; do NOT defer yet.** Applies to BOTH `--auto` and interactive (the human user-verify also needs the feature visible). Before calling a non-rendering feature `blocked`, check whether it's simply **gated off** ‚Äî a Firebase Remote Config flag, a feature toggle, an A/B bucket, or a `BuildConfig`/local default that a fetched server value overrode. **Signal:** the UI/behavior never appears **and** `logcat` shows the placement/feature was never even requested, while unrelated parts of the app work fine (e.g. other ads fill, other screens render) ‚Äî i.e. the gate resolved false, it's not a code failure. When that's the case, **force the gate to its enabled value using the project's own mechanism, as a LOCAL UNCOMMITTED change**, then rebuild the verify APK (`assembleAppDevDebug`) and re-run the verify:
> - **In-project flag source (preferred, self-serve):** flip the flag's resolved value at its source in the fix worktree ‚Äî set the local default true / add a debug-only override / short-circuit the remote fetch for that key (e.g. this project's `AdRemoteConfig` `booleanConfig("banner_pip", default=true)` values are overridden by fetched RC ‚Üí force them on for the debug build). Keep it a **test-only diff** (`git stash` it or a scratch edit), rebuild, verify, then **revert it BEFORE the `[1COMMIT]`** so the gate override NEVER enters the fix diff / PR. Since Gate 1's diff was already approved on the real fix, re-run Gate 1 only if the revert isn't clean.
> - **Remote-config console (when you can't/shouldn't touch code):** toggle the flag on for the debug app / a test condition in the Firebase Remote Config console, force a re-fetch on device (`am force-stop` + relaunch, or clear the RC cache), verify, then restore the flag. Requires console access ‚Äî if unavailable, use the in-project path.
>
> Note in the verify notes / PR body that the gate was force-enabled for observation and restored. **Only if the feature STILL can't be made to render after forcing the gate** ‚Üí defer as **`feature-gated`** (distinct from `adb-blocked`): the fix is code-correct but not device-observable in this environment ‚Äî record which flag/value QC must enable to verify, and (interactive) offer the dev the `feature-gated` defer vs. a code-review-only sign-off `[OPTIONS]`.

3. **Self-verify ‚Äî run the bundled verifier. For a UI-scoped ticket the design-anchored `android-ui-verify` is MANDATORY on this gate (interactive AND `--auto`); the behavioral `android-self-verify` runs ONLY under `--auto`.**
   - **[MANDATORY ‚Äî a smoke-check is NOT a pass] Whenever `android-ui-verify` runs it MUST execute the FULL design-anchored procedure, never a shortcut: (1) capture the device screen at the exact bug state, (2) fetch the screen's Figma node reference, (3) vision-diff device ‚ü∑ Figma across ALL 4 axes (layout/spacing ¬∑ color/theme ¬∑ typography ¬∑ alignment/integrity) with the per-axis table, AND (4) sweep the blast radius of every shared-UI symbol in the diff (token/dimens/style/shared composable/drawable ‚Üí the other screens that consume it). "The element is present / it looks right / the build compiled / I reasoned it through" is NOT a valid pass and is a BUG. A `pass` REQUIRES a SAVED device‚ü∑Figma comparison + the per-axis table + the blast-radius result; anything less is `fail`/`blocked`. This is the exact check that stops the "functionally fixed, visually wrong" and "the fix broke an adjacent screen" reopens ‚Äî the reason the gate exists. The verifier is dispatched with the Figma `node-id`(s) + acceptance tokens (Phase 3) + the diff so it CAN run all four axes + the sweep; a dispatch missing those is itself the bug.**
   - **`[TOKEN]` Dispatch the verifier as a SUBAGENT (Task tool), NOT inline on the main loop.** The verify drive Reads many heavy artifacts ‚Äî screencap PNGs, Figma reference images, `uiautomator` XML, `logcat` tails, `eyes_compare` output, the 5√ó auto-retries ‚Äî and those MUST stay in the **subagent's** context, never the main loop's. The subagent does all the `adb`/Figma/vision work in its own window and returns **ONLY the structured verdict block + the SAVED evidence file PATHS** (`.jira-bug/evidence/<TICKET>/‚Ä¶`) ‚Äî the main loop consumes that text and **never Reads a screenshot/XML/log itself**. (Same pattern as the Phase-3 analyzer fan-out and Phase-6 finalize ‚Äî verify was the one drive still running inline; this is the single biggest token sink, so isolating it is mandatory.) The subagent uses the device serial this phase already locked (step 1) but does NOT acquire/release the lock.
   - **Pick the verifier by ticket scope (Phase-3 scope):** the bug is about how the screen *looks* (layout/spacing, color/theme, typography, alignment, overflow/truncation, missing/misplaced element, wrong visual state) ‚Üí **`android-ui-verify`** (`references/android-ui-verify/SKILL.md`) ‚Äî design-anchored: it fetches the screen's Figma node, captures the device screen, **vision-diffs device ‚ü∑ Figma across 4 axes and BLOCKS on material deviation** (this is what stops the *"functionally fixed, visually wrong"* reopen), then **sweeps the blast radius** of any shared-UI change (theme/token/dimens/shared composable/drawable) to catch visual regressions on adjacent screens. Pass it the Figma `node-id`(s) + acceptance tokens from Phase 3, plus the diff (for the sweep). A behavioral+visual ticket ‚Üí run `android-ui-verify` then also confirm the behavior. ‚Äî A purely behavioral/logic bug ‚Üí **`android-self-verify`** (`references/android-self-verify/SKILL.md`).
   - **`--auto` set** ‚Üí dispatch the **chosen verifier** subagent (standalone fallback if the bundled copy is absent), passing: the **change summary** (ticket key + title), the **repro steps** (from the ticket/media), the **acceptance criteria** (from the approved plan + spec/figma; UI ‚Üí the Figma `node-id` + acceptance tokens), **variant** `appDev`/debug, the **already-locked `--serial`** AND the **`--owner-token <session-id>`** (step 1 ‚Äî the subagent drives every adb call via `device-lock.sh exec <serial> <token> --` and does NOT acquire/release the lock; it must never mint its own token nor fall back to bare `adb -s`), the **fix worktree dir** (so it verifies *this* change), and the **evidence dir** `.jira-bug/evidence/<TICKET>/` (created up front ‚Äî NOT inside the worktree, so evidence is never committed). The verifier launches that build, re-drives the exact repro + adjacent edge cases, and **captures SAVED evidence whose medium matches the bug** ‚Äî a **screenshot** for a static/visual bug, a **screen-recording video** (`adb shell screenrecord` ‚Üí `pull`) for a dynamic/flow/animation/scroll-jank/crash bug (both when in doubt) ‚Äî observes real pixels, watches `logcat` for exceptions/jank, and returns a structured `pass`/`fail`/`blocked` verdict **with the saved evidence file path(s)**. **A `pass` is invalid without a saved evidence file.** **Under `--auto` retry the whole verify drive up to 5√ó** (re-launch, re-navigate, re-capture ‚Äî try alternate content/data states each time, e.g. a different Book until one has a coin-locked next chapter) before returning `blocked`; never fail/blocked on the first miss. **`[AUTO]`: the verifier's verdict IS the verify gate (replaces the human user-verify card per `references/helpers/auto-run-test.md`).** Per ticket: `pass` (with saved evidence) ‚Üí **carry the evidence path(s) forward to Phase 6** (PR body + Jira) ‚Üí release device lock ‚Üí **Gate 3** (no human card) ¬∑ `fail` ‚Üí back to Phase 4 Fix-11 (counts against the 2√ó retry budget) ¬∑ `blocked` (state unreachable after the retries) ‚Üí **defer + report**, never claim a pass you couldn't observe, no human fallback in auto. **Blocked reason:** `adb-blocked` for an unreachable data/nav state; but if the feature is hidden by a gate, first apply **`[FEATURE-GATE]`** (force the flag on, rebuild, re-verify) ‚Äî only if it still won't render ‚Üí defer as **`feature-gated`** (record the flag/value QC must enable).
   - **default (no `--auto`)** ‚Üí for a **UI-scoped** ticket, STILL run the design-anchored `android-ui-verify` (the FULL procedure above) as a mandatory automated design check whose per-axis table + saved evidence feed the human user-verify card ‚Äî a material Figma deviation or a blast-radius regression bounces the fix to Phase 4 *before* the human is even asked. For a **purely behavioral** ticket, skip `android-self-verify` and go straight to user-verify. (Interactive still keeps the human user-verify card as the final sign-off; the design-anchored pass does not replace it, it precedes and informs it.)

   - **üé´ Ticket:** `<TICKET-KEY>` ‚Äî <link to the Jira ticket>
   - **Ti√™u ƒë·ªÅ:** <ticket title>
   - **M√¥ t·∫£:** <short description of the bug / the fix>
   - **·∫¢nh/Video:** <the ticket's attachment image(s)/video ‚Äî embed/inline so the dev sees the original repro>

   - **C√°c b∆∞·ªõc ki·ªÉm tra:** <numbered steps the dev follows on-device to verify>
   
   `Needs update` ‚Üí decide scope as in Gate 1 (Phase 4 impl loop **or** Phase 3 plan loop) ‚Üí re-enter from Gate 1. `Verified OK` ‚Üí release the device lock ‚Üí **Gate 3**.

### Gate 3 ‚Äî Create/update the PR + Jira finalize (hand to Phase 6)
On **user-verify pass** ‚Üí hand to **Phase 6** (`references/phase6-commit-pr.md`): the ticket's `[1COMMIT]` is already on the batch branch (merged at Gate 1 `OK`), so **open the batch PR if it doesn't exist yet, else just let the new commit update the existing batch PR**, then **attach the PR link to the Jira ticket and transition it ‚Üí Resolved in a BACKGROUND subagent** (so the main session isn't blocked on the Jira round-trips), log work, arm the watcher (once per PR). The **PR description carries ALL Phase-3 + Phase-5 info per ticket** (root cause, scorecard, plan, files changed, verification steps + evidence, expected result) ‚Äî see the PR-description template in Phase 6.

## Device driver ‚Äî **`adb` only, NOT mobile-mcp** (runs INSIDE the verifier subagent)
Gate 2's `--auto` self-verify runs through the chosen verifier skill **dispatched as a subagent** (`[TOKEN]`, step 3 ‚Äî `android-ui-verify` / `android-self-verify`); that skill (and any manual drive) interacts **only via `adb`** against the locked serial ‚Äî **never mobile-mcp**. **`[TOKEN]` All the Read-heavy verbs below execute in the SUBAGENT's context ‚Äî keep them out of the main loop, and minimize even there:** Read only the **decisive** screenshot(s) (the bug state + the after-fix result), NOT every intermediate frame; **never `cat` a full `uiautomator` XML into context** ‚Äî `grep` the one node's `bounds` and discard the rest; cap `logcat` at `-t 100`. **Every verb below MUST run through the lock gate ‚Äî `LOCK=<skill-dir>/assets/device-lock.sh`, `bash "$LOCK" exec <SERIAL> "$TOKEN" -- <adb args>` ‚Äî where `$TOKEN` is the raw session id the parent passed in (`--owner-token`). A bare `adb -s <SERIAL> ‚Ä¶` is a BUG: it bypasses the lock, so a second terminal drives the same device and you capture ITS screen as your evidence (`references/helpers/device-lock.md`). `NOTOWNER` (exit 3) ‚Üí stop, never fall back to raw adb.** Core verbs (shown post-`exec --`):
- **Install/launch:** `bash "$LOCK" install <SERIAL> "$TOKEN" <apk>` (never a bare `adb install`) ¬∑ `exec ‚Ä¶ -- shell am start -n <pkg>/<launcher>` (or `shell monkey -p <pkg> -c android.intent.category.LAUNCHER 1`) ¬∑ `am force-stop <pkg>` to reset.
- **Save evidence (medium by bug type):** static/visual bug ‚Üí keep the screencap PNG as the evidence file; dynamic/flow/animation/scroll/crash bug ‚Üí record a video: `bash "$LOCK" exec <SERIAL> "$TOKEN" -- shell screenrecord --time-limit 30 /sdcard/<TICKET>.mp4` (run in background while you drive the repro) ‚Üí `bash "$LOCK" exec <SERIAL> "$TOKEN" -- pull /sdcard/<TICKET>.mp4 .jira-bug/evidence/<TICKET>/` ‚Üí `bash "$LOCK" exec <SERIAL> "$TOKEN" -- shell rm /sdcard/<TICKET>.mp4`. Save under `.jira-bug/evidence/<TICKET>/` (never the worktree).
- **See the screen (PRIMARY observation = real pixels):** `bash "$LOCK" exec <SERIAL> "$TOKEN" -- exec-out screencap -p > <scratchpad>/shot.png`, then **understand it via `[MEDIA]` human-mcp `eyes_analyze({ source: "<png>", focus: "..." })`** (text back, no bytes in context; `references/helpers/media-analysis.md`). **Fallback** (human-mcp exhausted-today / unconfigured) ‚Üí **Read the PNG** natively (then `[TOKEN]` resize navigation-only shots ‚â§512px per `references/helpers/media-preprocessing.md`; decisive states full-res). ‚öÝÔ∏è **Screencap caveat (load-bearing):** interstitial **ad / onboarding / secure surfaces capture BLACK** ‚Äî when a frame is all-black, it's almost always an ad/secure screen, NOT a real failure: press **BACK** (`input keyevent 4`) to dismiss the interstitial, then re-capture the actual app screen. Real **app content (reader, lists, bottom-sheets) captures fine**, so observe behaviour by screenshot. On Git Bash, prefix the adb command with `MSYS_NO_PATHCONV=1` so `/sdcard/...` isn't mangled to a Windows path, and `screencap -p /sdcard/s.png` + `pull` if the `exec-out` pipe yields 0 bytes.
- **`uiautomator` is for TAP TARGETS ONLY, never the sole observation:** `uiautomator dump /sdcard/v.xml && exec-out cat /sdcard/v.xml | grep -o 'text="<label>"[^>]*bounds="[^"]*"'` ‚Üí parse `bounds="[x1,y1][x2,y2]"`, tap the center. **Never pipe the whole XML into context** (`[TOKEN]`) ‚Äî grep the one node you need. It **cannot read reader/WebView/Compose-canvas text** ‚Äî a static uiautomator dump while content scrolls is NOT evidence the screen didn't change; confirm by screenshot.
- **Interact:** `input tap <x> <y>` ¬∑ `input swipe <x1> <y1> <x2> <y2> <ms>` (a vertical reader scrolls with swipe-up **away from the right-edge scrollbar**, e.g. `x‚âà450`; slow ~600ms exposes jank) ¬∑ `input text '<str>'` ¬∑ `input keyevent <KEY>` (BACK=4).
- **Logs:** `bash "$LOCK" exec <SERIAL> "$TOKEN" -- logcat -c` before a run, then `‚Ä¶ -- logcat -d -t 100` (filter pkg / `AndroidRuntime`; raise the count only to chase a specific trace ‚Äî `[TOKEN]`). Behaviour signals (e.g. `reportReachedEnd`/unlock/coin events) confirm a step actually fired.
- App package from the project (`applicationId`, e.g. `com.readbook.bookstore.audiobooks`); pick the `appDev`/debug flavor.
- **Reach the right data state:** the repro often needs specific content (e.g. a Book whose **next chapter is coin-locked** ‚Äî a `Gi√° 0`/free book won't trigger the unlock sheet). If no suitable content/account state is reachable, that's `blocked`, not `pass`.

> The emulator/device may lack network, or be stuck behind ads/onboarding. If the app can't reach the screen under test (or the required data state) after the skill's **5** retries, the `--auto` self-verify returns **`blocked`** ‚Äî never claim a pass you couldn't observe. The human may waive the device step (`[OPTIONS]`). **If the screen under test never renders because a remote-config / feature flag gates it off (not a nav/data-state problem), apply `[FEATURE-GATE]` in Gate 2 first ‚Äî force the flag on for the debug build, verify, revert ‚Äî before falling back to a `feature-gated` defer.**

## Granularity
- **Single** ‚Üí this phase runs for the one ticket (show+auto-open its diff ‚Üí install its APK ‚Üí user-verify ‚Üí PR).
- **Multi/team** ‚Üí **per ticket, streamed ‚Äî NOT a windowed round.** Each ticket, the instant its fixer is green: Gate 1 shows its diff + auto-opens the browser ‚Üí `OK` ‚Üí **merge it into the batch branch first, immediately** ‚Üí Gate 2 installs its APK + one user-verify block ‚Üí Phase 6 opens/updates the ONE batch PR carrying that ticket's `[1COMMIT]` (never `--squash`). **Never wait for another ticket.** A ticket that fails review/verify loops back to its own fixer (Phase 4/Phase 3 by scope) without blocking the others (Per-ticket review & batch-merge block, `references/run-blocks.md`).

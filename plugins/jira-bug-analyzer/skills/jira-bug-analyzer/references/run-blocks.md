# Shared building blocks

> Referenced by `references/phase3-analyze.md`, `references/phase4-fix-and-build.md` and the per-mode init files (`references/phase1-init-single-mode.md`, `references/phase1-init-multi-mode-without-team.md`, `references/phase1-init-multi-mode-with-team.md`). Cross-cutting invariants (`[REST]` `[VN]` `[OPTIONS]` `[CLAIM]` `[VERIFY]` `[1COMMIT]`) are defined ONCE in SKILL.md **Golden Rules** — cited by tag here, not restated.

### Post-analysis dedup (status memory) — CHECK AFTER ANALYZING, never before claim
Bugs are pulled from `Request`/`Reopened` (Init-9 default) — not yet owned by any dev — so there is **no pre-claim dedup gate**: don't scan the ledger before picking/claiming. Dedup runs **only once an analyzer has settled a root cause** (the first moment a real `root_cause_slug` exists). As the analyzer settles the root cause (and as the orchestrator processes its return — Multi fan-out / Fix-6), read the consolidated ledger **`project/<PROJ>/<PHASE>/session/record_bug_status.md`** (glob `project/<PROJ>/*/session/record_bug_status.md` to catch a prior-phase fix — `memory-sync.sh path`; the Init `pull 600` copy suffices — only `--resume` forces `pull 0`). One file holds every analyzed bug's row. Detect:
- **Same ticket** already `analyzing`/`analyzed`/`fixing`/`pr-created`/`done` by another dev or a prior session, OR
- **Another ticket with the SAME `root_cause_slug`** in-flight/done (duplicate root cause — many bugs share one fix).

The dedup match is a judgment call → run on **opus** (Init-6). When an analyzer returns `duplicate` (it matched an existing `root_cause_slug` in the ledger), **INFORM the dev explicitly `[VN]` before any fix** — name the **other dev** (the row's `dev`), **which ticket** they own, the **shared `root_cause_slug`**, and their **stage** (`analyzing`/`fixing`/`pr-created`/`done`): e.g. *"⚠️ AIP686-200 cùng root cause `comment/cms-fake-reviews-never-loaded` với AIP686-197 — huypq đang fix (fixing). Tránh fix trùng."* — then ask `[OPTIONS]` (`Skip — đã/đang xử lý` / `Vẫn làm`). Skip → mark `commented`/closed, no fixer. (The `record_bug_status.md` ledger is the dedup signal, upserted by `memory-keeper` the moment each bug is analyzed; `[CLAIM]` is only the ownership stamp at pick — it is not a dedup step.)

### Source-of-truth capture → its own phase (background, non-blocking)
The Source-of-truth capture is **Phase 2** — full spec (mandatory *ask* for spec/figma · background capture of one `<slug>.md` per external source + `metadata.json` manifest · external-source direct read, pending-if-unreadable re-tried each run, **no request folder** · `capturing`/`ready`/`built`/`blocked` returns · MCP re-probe) lives in **`references/phase2-source-of-truth.md`**. The mandatory ask precedes any claim/analyzer; the digest capture runs in the **background** (it no longer blocks). It captures the **full phase content, independent of the picked tickets**. Analyze/Fix resolve SOT via the Context-source rule (memory if landed, else the dev's provided refs — `references/phase3-analyze.md`). Not restated here.

### In-Progress lock
Apply `[CLAIM]` the moment a ticket is picked (before Fix-3) — atomic transition→In Progress + assign to the dev account. Mechanics: `references/helpers/jira-rest-api.md`. If the requested pull status *is* `In Progress`, the claim is a status no-op (dedup relies on the batch state file) — still confirm the assignee.

### Pre-fix context import (ground truth)
Before implementing, read the source-of-truth for the affected screen via the **Context-source rule** (`references/phase3-analyze.md`): the digests if the background capture has landed (`project/<PROJ>/<PHASE>/doc/` — `spec.md`, `figma.md`, one `<slug>.md` per external source; `metadata.json` indexes their status), **else the dev's provided spec/figma refs** from Phase 2. Either way these are ground truth, not the repo's `CLAUDE.md`. When reading the digests: spec section (behavior+copy), the relevant figma `node-id`, the relevant external source's `<slug>.md`. A specific ref missing inside a present digest → note + proceed. If a needed external source's `<slug>.md`/`metadata.json` is still `pending` (a Google Sheet/Drive value that couldn't be read directly), **do not invent that data** — implement what the accessible spec pins down, flag the dependent part as awaiting a readable source (or the dev pasting it), and surface it in the plan/verify so it isn't silently assumed.

### Fix worktree (isolation) — every fix
Every fix runs in an **isolated git worktree** (never disturbs the user's main tree). Create at branch time (Fix-10): **ALWAYS `git fetch origin` fresh first to get the NEWEST `origin/<BASE>`** (never branch off a stale local ref), then branch off the latest **resolved base branch `origin/<BASE>`** (the project's integration branch — see "Base branch" below; e.g. `develop` for AIP686, `main` for others — NOT a hardcoded `origin/main`). This is the SAME ref Phase 3 grounded its analysis on, so the diff stays minimal and the `file:line`s line up. `phaseX` = the **phase resolved at the Source-of-truth capture** (`@N` arg or spec-title-derived — SKILL.md Phase resolution; e.g. `phase3`), never a blind `phase1`:
```bash
git fetch origin
git worktree add -b phaseX/fixbug/<Ticket-Id> ../learnova-<Ticket-Id> origin/<BASE>          # single
git worktree add -b phaseX/fixbug/batch-<YYMMDD> ../learnova-batch-<YYMMDD> origin/<BASE>      # batch
# branch exists → omit -b, pass the existing branch (rebase onto origin/<BASE> if stale)

# [MANDATORY right after EVERY worktree add] seed the gitignored build-config files a fresh
# worktree does NOT inherit — without local.properties the Gradle build dies with
# "SDK location not found ... local.properties". Copy them from the main repo (the run's CWD/root):
for f in local.properties keystore.properties signing.properties; do
  [ -f "<root>/$f" ] && cp "<root>/$f" "<worktree-dir>/$f"
done
# (also copy any other untracked-but-build-required file the project needs, e.g. google-services.json
#  ONLY if it is gitignored AND absent in the worktree — never echo/commit these secret files.)
```
All edits/builds/app-run happen in that dir. **A worktree is a fresh checkout — gitignored files (`local.properties`, keystores, `*.properties` secrets) are NOT carried over, so the copy step above is REQUIRED before the first `./gradlew` in any new worktree (single, batch, fixer, or the verify/build worktree). Skipping it = `SDK location not found` on the very first build.** **Keep it until the PR merges/closes** (the watcher needs it for review fixes; it removes it on merge/close). The `worktree` skill can manage create/cleanup. The **PR base is the SAME `<BASE>`** (`gh pr create --base <BASE>`); rounds are cherry-picked/rebased onto `origin/<BASE>`. A worktree mistakenly cut from `origin/main` when `<BASE>` is `develop` → rebase it onto `origin/<BASE>` before building/PR (the divergence otherwise pollutes the diff).

### Base branch (resolve ONCE at init, persist in setup.json, reuse for every worktree + PR)
The integration branch to branch from and PR into is **per-project, not always `main`** (CLAUDE.md may mandate `develop`, `dev`, etc.). Resolve it during the Init phase and cache it so other devs/sessions reuse the same answer — never re-guess per fix:
- **setup.json key `baseBranch`** (peer of `board`/`pullQuery`) is the source of truth. **Read order = LOCAL `.jira-bug/setup.json` first → else the shared remote `project/<KEY>/setup.json`.** A value saved in **either** layer is authoritative config → **AUTO-REUSE it verbatim as `<BASE>`, NO prompt** (init already adopts a remote-only value and hydrates the local mirror, so the next run is a pure-local hit).
- **`BASEBRANCH=none` = absent from BOTH layers (local AND the shared remote mirror) — a genuine first-time config, NOT a saved value being ignored.** Only then ASK the dev: **MANDATORY `AskUserQuestion`, do NOT proceed (no worktree, no PR) until answered.** A value resolved from heuristics (CLAUDE.md / git default) is only the **pre-filled recommended option**, never auto-applied — saved memory auto-reuses, a guess still asks:
  - Build the options by resolving candidates in order: (1) `BASEBRANCH_SUGGEST` from INIT-STATUS (the shared `project/<KEY>/setup.json` value, if init emitted one); (2) the repo's `CLAUDE.md` / contributing docs PR target (e.g. *"Branch from / PR into `develop`"*); (3) the repo's default branch on the remote (`gh repo view --json defaultBranchRef -q .defaultBranchRef.name` or `git symbolic-ref refs/remotes/origin/HEAD`); (4) the branch currently checked out if it's a long-lived integration branch. Offer the **best candidate as the recommended first option** (label it *(recommended)*), list the other distinct candidates, and **always include an `Other` free-text** so the dev can type any branch. If CLAUDE.md and the remote default disagree, surface the mismatch in the question text.
  - **Only after the dev picks**, persist their choice to **BOTH layers — this is two writes, not one. The shared push is NOT optional and is NOT deferred.**
    1. **Local mirror:** `bash <skill-dir>/assets/setup-json.sh merge-set "<root>/.jira-bug/setup.json" baseBranch=<branch>`.
    2. **Shared memory (cross-dev) — MUST happen the same turn:** merge `baseBranch` into the clone's `project/<KEY>/setup.json` (stamp `savedBy`/`savedAt`) and push. **ALWAYS write via `setup-json.sh merge-set` — NEVER hand-write the JSON** (a hand-written trailing comma is what silently broke remote-config reuse). Do it via the `memory-keeper` agent (`action: project-setup`), or inline:
       ```bash
       CLONE=$(bash <skill-dir>/assets/memory-sync.sh path)
       bash <skill-dir>/assets/memory-sync.sh pull
       bash <skill-dir>/assets/setup-json.sh merge-set "$CLONE/project/<KEY>/setup.json" \
            baseBranch=<branch> savedBy=<dev> savedAt=<iso8601>   # canonical, trailing-comma-safe, keeps other fields
       bash <skill-dir>/assets/memory-sync.sh push "project/<KEY>/setup.json" "chore(<KEY>): set baseBranch=<branch>"
       ```
    A local-only write is a BUG — the next dev/session won't inherit it and will be re-asked. Resolving a previously-absent `baseBranch` is a `project-setup` change (see `references/helpers/memory.md`), exactly like first-run config.
- **Do NOT treat a confident resolution as permission to skip the ask.** `BASEBRANCH_SUGGEST=develop` means "default the prompt to develop", NOT "use develop without asking". The dev must tap an option (or type one) every time `baseBranch` is absent.

### Multi fan-out / batch parallelism (per-ticket PIPELINE: analyze → approve → fix — NO barrier)
Multi's per-turn fixing machinery. Same ordering rule as single: **plan approved BEFORE any code — never fix-first.** Each ticket flows analyze → present → approve → fix **independently and concurrently**. Only fixers touch the filesystem (each in its own worktree). Interactive gates stay in the main session.

> **NO BARRIER — stream sequentially via a FIFO completion queue + single interactive floor.** Dispatch all analyzers as **background agents in one message** (NOT foreground-parallel — foreground-parallel returns only after the *last* finishes, which is the "waited for all" bug). As each completion notification arrives, append the ticket to a **ready-to-present queue (FIFO by completion order)**. Only ONE ticket holds the **interactive floor** at a time: present the queue head, take its one disposition, then release. On `Approve` its fixer launches **in the background** right away — in parallel with other analyzers still running and other fixers already fixing. **While the floor is held, later completions just enqueue — never interrupt.** Never wait to collect all plans, never wait for all approvals. Full step-by-step: `references/phase3-analyze.md` → "Multi mode — analyze fan-out".

**Analyzer agents** (one per claimed ticket, all dispatched in one message as **BACKGROUND agents — `run_in_background: true`** so each completion notifies independently — READ-ONLY, `model: opus`): run Fix-3–Fix-6 silently. MUST NOT create a worktree, edit files, or write Jira. Receives (explicit prompt, no session history): ticket key (already claimed), project root, **the resolved base branch `<BASE>`**, the phase source-of-truth digests, **the `record_bug_status.md` ledger** (so it can match an existing `root_cause_slug` and NOT re-fix a root cause another ticket already handled — call out the dup in its return instead), and the Fix-6 output discipline + `[VN]` plan format + pre-fix context rules. **It MUST `git fetch origin` first and analyze the code as it exists on the latest `origin/<BASE>`** (read-only — `git grep`/`git show` against that ref; never check out / edit / worktree), so its `file:line` references match the ref Phase 4 will branch from — never the stale local working tree. **It MUST apply the Decide-from-SOT rule (`references/phase3-analyze.md`): resolve the plan from the SOT (digests if landed, else the dev's provided refs) and only emit a `Câu hỏi cần làm rõ` when the answer is genuinely absent from both the SOT and the dev input, or the sources contradict / the logic is weird — never ask what the spec/figma/external source already answers.** Returns its final message as raw data: `status` = `plan-ready` (+ Root cause + Fix plan block, Fix-6 format, incl. the `root_cause_slug` it settled on) | `duplicate` (same `root_cause_slug` as an in-flight/done ticket → name it, no new fix) | `comment-needed` (non-code → Fix-9 reason + drafted `[VN]` comment + tech detail) | `blocked` (what's missing). Each completes at a different time — handle each on arrival.

> **After each analyzer returns** (before/with presenting the plan): the orchestrator has `memory-keeper` **upsert that ticket's row into `record_bug_status.md`** (`status: analyzed`, with the `root_cause_slug` + the concise root cause in `summary` + touched `file:line`s in `files`) so the next analyzer's dedup sees this bug. **This ledger row is the ONLY memory record — there is no per-ticket file.** The full Fix-6 block (Fix plan / Risk / clarify / ticket meta) stays in chat, never in memory.
>
> **[MANDATORY — push the ledger row to the SHARED REMOTE the moment it's written, never local-only.** Cross-dev / cross-session dedup *and* the root cause only reach teammates if the row lands in the shared memory repo, so the write is incomplete until it is **committed AND pushed to `origin`** (via `memory-keeper` / `assets/memory-sync.sh push`). A row that lives only in the local mirror is invisible to other devs and lets two people fix the same bug. **Dispatch the commit+push as a BACKGROUND `memory-keeper` agent (`[BGMEM]`) immediately after each analyzer settles its root cause** — same turn, non-blocking; do NOT batch it to end-of-run, do NOT wait for the fix/PR, and do NOT hold the interactive floor on the push (its result is consumed lazily at the next dedup/claim gate). The row MUST be re-pushed on every transition (`analyzed` → `fixing` → `fixed`/`ready` → `done`/`commented`), keeping its `summary`/`files` in sync with any plan revision. If the push fails, surface it — un-pushed memory means dedup is silently off and the root cause never reaches the team.]**

**Per-ticket approval hinge** (main session, holds the single floor for ONE exchange per ticket, then releases): present the FIFO-queue head's `plan-ready` block — the **FULL Fix-6 block VERBATIM, every field, never trimmed/summarized; every file ref = short-display markdown link `[File.kt:line](full/repo-relative/path:line)` (short text, FULL path in the target)** (mandatory rule in `references/phase3-analyze.md`) — `[OPTIONS]` (`Approve`/`Needs changes`). On **Approve** → mark `approved` → launch that ticket's fixer **in the background** → **release the floor** (present the next queued head). On **Needs changes** → take dev direction → **re-dispatch the revised analyzer in the background** → **release the floor**; the revised plan **re-enters the FIFO queue when it lands** (dev moves on to the next ready ticket, returns to this one when it re-surfaces — does not block on the revise). `comment-needed`/`blocked` → resolve that one prompt (VN comment / ask) → release the floor; never gets a fixer.

**Fixer agents** (one per APPROVED ticket, launched on approval — `model: sonnet`): receives its approved plan verbatim + context + worktree convention; creates its worktree, implements per the plan (Fix-11 logic), compile-checks. On clean compile → run the **app build to verify compilation** (`./gradlew assembleAppDevDebug` in its worktree — Fix-12), which doubles as the **build pre-warm** (below); a green APK = the fixer's done gate, a red build loops back to fixing. Does NOT verify/review/commit/PR or post Jira. Returns `fixed-in-worktree` (path + branch + files + diff summary + compile result) | `blocked`. **Plan wrong mid-fix → STOP, return `blocked` with the finding — never silently deviate**; dev re-approves a revised plan first. Fixers run parallel with each other and with still-running analyzers.

> **[HARD RULE — ALL source edits happen INSIDE a fixer agent (background, in its own worktree). The orchestrator / main-loop session NEVER edits source files itself — not even a "trivial" one-line reorder, rename, padding tweak, or conflict-driven touch.]** Every code change is delegated; the main loop only *coordinates* (presents plans, takes dispositions, dispatches agents, runs git/gh/adb glue, drives the interactive gates). This applies to the FIRST fix **and to every request-change after it**:
> - **Request-change with IMPL scope** (Gate-1 diff "Needs change", on-device user-verify FAIL, or any "tweak the code" the dev asks for — including reorder/styling/padding/copy) → **re-dispatch THAT ticket's fixer as a BACKGROUND agent** (`run_in_background: true`, sonnet, its existing worktree) handed the dev's exact direction + the file:line scope; it edits in the worktree, rebuilds, returns `fixed-in-worktree`. **Do NOT open the file and Edit it on the main loop.** The orchestrator then re-runs the diff/verify gates on the agent's result.
> - **Request-change with PLAN scope** (the approach itself is wrong) → re-dispatch the **analyzer** in the background (per the approval-hinge rule), get a revised plan, re-approve, then a fixer applies it.
> - The ONLY edits the main loop performs directly are **non-source git mechanics** that are not authored changes: resolving a cherry-pick/merge **conflict** during batch-merge (a merge resolution, not a code edit), and writing untracked run-state/config (`.jira-bug/*`, copying `local.properties`). Anything that changes the meaning of a `.kt`/`.xml`/source file = a fixer agent, always.
> - **Why:** main-loop edits bypass the worktree isolation, the fixer's compile/build gate, and the per-ticket commit discipline — and they freeze the interactive loop on one ticket. Delegating keeps every authored change isolated, built, and attributable, and lets other tickets keep streaming.

**Multi turn-state file** `<repo>/.jira-bug/batch-progress.md` (untracked; recovery anchor): model, board, picked tickets with per-ticket status (`pending`/`analyzing`/`plan-ready`/`approved`/`fixing`/`ready`/`done`/`commented`/`blocked`), each worktree+branch, spec/figma links. Update as agents are dispatched/return and gates pass. (Single mode = one bug, no agents, no state file — run the single workflow inline.)

### Post-fix gates → moved to Phase 5 (Verify) + Phase 6 (Commit & PR): per-ticket diff review (auto-open browser) → merge to batch branch first → install APK → user-verify (`--auto` → the `android-self-verify` skill, 5× retries) → commit → PR (Phase 6, a background agent under `--auto`) → Jira; then the `--auto` loop returns to Phase 1
After Phase 4 fixes + **builds the APK**, the review/verify/commit/PR/Jira gates are **separate phases** — **`references/phase5-verify.md`** (verify) + **`references/phase6-commit-pr.md`** (commit & PR). `[VERIFY]` — never PR before the diff review (OK) + user-verify pass. **Per-ticket, the instant the fixer returns green — never wait for other tickets** (Per-ticket review & batch-merge block). **Phase 5 order:** (1) **show the diff RIGHT AWAY and auto-open the dev's browser** on it → `OK`/`Needs change` (needs change → Phase 4 for impl scope, Phase 3 for plan scope — that one ticket only); (2) **`OK` → merge that ticket into the batch branch FIRST, immediately** (cherry-pick its `[1COMMIT]` onto `phaseX/fixbug/batch-<YYMMDD>`), then **install its APK** (`adb install -r -d`) + user-verify card; **adb self-verify runs ONLY on `--auto`** (driven by `adb` — `am start` / `input tap|swipe|text` / `uiautomator dump` / `exec-out screencap` / `logcat` — NOT mobile-mcp); (3) hand to Phase 6. The full gate sequence + per-gate loop-backs live in Phase 5; the **device lock** (`references/helpers/device-lock.md`) still applies. **`[AUTO]`: under `--auto` these human gates are replaced — `code-reviewer` gates the diff and adb self-verify is the verify gate (no user-verify card), per `references/helpers/auto-run-test.md`.**

> **Reuse the Phase-4 APK:** Phase 4 already ran `assembleAppDevDebug`, so Phase 5 **installs that APK** rather than rebuilding. Only rebuild if Gate 1's review loop changed the code (then re-assemble that worktree). Never start a second concurrent build of the same worktree.

> **Granularity:** single → Phase 5 runs per the one ticket (show+auto-open its diff → install its APK → user-verify → Phase 6 PR). Multi/team → **per ticket, streamed** (NOT a windowed round): each ready ticket gets its own immediate diff review (auto-open browser) → merge into the batch branch first → install its APK + user-verify → its commit/PR-update. Never block one ticket on another.

### Per-ticket review & batch-merge (multi/team) — stream each ticket, NEVER window-wait
Each ticket ships **the moment its fixer returns green** — do NOT wait for other tickets, do NOT batch a time window, do NOT defer review until a round cutoff. Review is per ticket, in completion order.
- **Batch branch (the batch's integration / "main" branch):** the FIRST ticket to turn `ready` creates ONE batch branch `phaseX/fixbug/batch-<YYMMDD>` off `origin/<BASE>` (the resolved `setup.json.baseBranch`, e.g. `develop` — NOT a hardcoded `main`). This is the **"main branch of the batch"** every approved ticket merges into. Record it (`batchBranch`) in `.jira-bug/batch-progress.md`.
- **Per ticket, on `fixed-in-worktree` (green build) — IMMEDIATELY, in arrival order (one interactive floor at a time; other fixers keep running):**
  1. **Show the diff RIGHT AWAY + auto-open the browser** (Phase 5 Gate 1). Push the ticket's fix branch, then **actually launch the dev's browser** on its diff — `gh pr diff <pr> --web` if a PR/draft exists, else open the compare URL `https://github.com/<owner>/<repo>/compare/<BASE>...<branch>?expand=1` via the OS opener (`start "" "<url>"` on Windows / `python -m webbrowser "<url>"` / `xdg-open` / `open`). Do NOT merely print the link — open it.
  2. **Ask the dev how the diff is** `[OPTIONS]` (`OK`/`Needs change`). `Needs change` → loop **that one ticket** back by scope **by re-dispatching a background agent — NEVER by editing the file on the main loop** (HARD RULE under "Fixer agents" above): impl tweak → **re-dispatch its fixer** (sonnet, its worktree, handed the dev's exact direction) which edits + rebuilds + returns `fixed-in-worktree`; plan-level → re-dispatch the analyzer (Phase 3). Then re-show the rebuilt diff; **other tickets are untouched and keep streaming**. This is the same rule for an on-device user-verify FAIL at Gate 2 — the re-fix is a fixer agent, not a main-loop edit.
  3. **OK → merge THAT ticket into the batch branch FIRST, immediately** — cherry-pick its one `[1COMMIT]` onto `phaseX/fixbug/batch-<YYMMDD>` (preserve the per-ticket commit; a cherry-pick conflict → rebase the ticket branch onto the current batch tip, re-build, retry). **Never wait for another ticket to do this merge.** Then carry that ticket through Phase 5 Gate 2 on-device **user-verify** (`[VERIFY]`, per ticket; `--auto` → adb self-verify) and on pass hand to **Phase 6** to open/update the batch PR + finalize Jira for that ticket.
- **`[AUTO]`:** the `OK`/`Needs change` diff gate becomes the `code-reviewer` gate (`auto-run-test.md`) — no-critical → auto-merge into the batch branch + continue; a **critical** finding → re-fix (counts against the retry budget); `verify`/`adb-blocked`/retries-exhausted → defer that ticket + report. No human diff prompt, no human user-verify card.
- **PR:** ONE PR per batch, targeting `<BASE>` from the batch branch (`gh pr create --base <BASE>`, full Phase-3+5 body), **opened on the first merged+verified ticket and updated as later tickets merge in** (each carries its own `[1COMMIT]` — never squash-merge). Arm one watcher on it.

### PR watcher (surface review comments + clean up worktree)
> **Ownership note:** this watcher (review-comment observation/fixing + worktree cleanup) is **acted on by `--manager`** (`references/phase-manager-mode.md`), which also adds KB-backfill (Job C). Phase 6 only *arms* it by writing `.jira-bug/pr-watch.json` at PR creation. **Under `--auto` the looper opens the PR (Phase 6, background) + writes the watch file, then moves on** — run `--manager` after the batch to reconcile. The block below describes the watch-file shape + interactive arming; the reconcile loop that consumes it lives in `phase-manager-mode.md`.

Ticket is already **Resolved at PR creation** — the watcher does NOT resolve/worklog/comment on merge. Two jobs per watched PR: **(1)** while open → detect new review/line/issue comments → notify → draft a fix per comment, apply only on the dev's approval (never auto-edit unattended); **(2)** on merge/close → remove the fix worktree.

> **`CronCreate` is not a background daemon** — jobs fire only while a Claude session is open+idle and die when Claude exits. The durable, load-bearing arm is **writing the state file `.jira-bug/pr-watch.json`** (incl. `commentsCursor`), not the cron. Triggers: **(A)** the 1-min cron while the session stays open (`durable: true`, fires `/jira-bug-analyzer --manager <KEY>` so it reconciles); **(B)** a `--manager` run — the durable path that survives session death (also the minimal Init-5 worktree cleanup on any skill start). Comments left while Claude is closed surface at the next `--manager` run, never in the background — never promise unattended surfacing. Full mechanism (verified arm, per-PR lock, comment poll + interactive flow, worktree cleanup): `references/helpers/pr-merge-watcher.md`. Arm on PR creation `[OPTIONS]` (`Watch` default / `Don't watch`).

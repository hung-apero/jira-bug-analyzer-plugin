# Phase 3 — Analyze (claim → investigate → root-cause + plan → approval)

> The flow's third phase: **Intake (phase1) → Source-of-truth capture (phase2, background) → Analyze (THIS phase) → Fix (phase4).** Analysis no longer waits on the source-of-truth digest — Phase 2 only *launched* the capture in the background. This phase claims the ticket, investigates, converges on a root cause, presents the fix plan, and gets approval. Approved plans hand off to **Phase 4** (`references/phase4-fix-and-build.md`). Shared machinery lives in `references/run-blocks.md`; invariants are SKILL.md Golden Rules (`[CLAIM]` `[VERIFY]` `[1COMMIT]` `[VN]` `[OPTIONS]` `[REST]`). Confidence rubric: `references/helpers/confidence-rubric.md`.

## Context-source rule (memory-if-ready, else the dev's provided input)
This phase (and Phase 4) need the source-of-truth as ground truth — but Phase 2 runs the capture in the **background**, so the digest may not be in memory yet. Resolve the SOT context in this order, per ticket, at the moment it's needed:
1. **Memory ready** — the background `source-of-truth` capture has landed (`project/<PROJ>/<PHASE>/doc/metadata.json` present + the relevant digests written — `spec.md`, `figma.md`, the external source's `<slug>.md`) → **use the digests** (the richer, distilled SOT). This is the preferred source. **`[RAG]` Retrieve by SECTION, never read the whole digest:** read `metadata.json`'s `sections` index (or the digest's top-of-file TOC) → match the ticket's screen/feature to its `<anchor-slug>` → read **ONLY that `## <anchor-slug>` section** of `spec.md`/`figma.md` (grep the anchor + read its span). Read the whole digest only if no section matches and you must scan the TOC text. This is the token-saving payoff of the structured digest — one section per ticket, not the full spec.
2. **Memory not ready yet** — capture still running → **fall back to the dev's provided spec/figma/other refs** captured at Phase 2 (the links / pasted content the dev supplied up front). Because spec + figma are **mandatory to provide**, there is always *some* context — never block waiting for the digest.
3. **Prefer memory once it lands** — if the digest finishes mid-analysis (or by the time Phase 4 fixes), switch to it; it supersedes the raw refs. No separate blocking re-check gate — just use the better source when available, and flag any contradiction between the raw ref and the landed digest to the dev.

Do **not** stall the claim/fetch/analyze waiting on the background capture. The only thing that ever blocks is the Phase-2 *ask* for the mandatory spec/figma links (already done before this phase).

## Decide-from-SOT rule (resolve the plan from ground truth — ask the dev ONLY for genuine gaps)
The analyzer's job is to **decide** the fix plan, not to offload design decisions back to the dev. Before writing any `Câu hỏi cần làm rõ`:
1. **Answer it from the SOT first.** Look in the resolved SOT (the landed digests if ready — `spec.md`, `figma.md`, the relevant external `<slug>.md`; **else the dev's provided spec/figma/other refs** from Phase 2). If the spec/figma/Sheet pins down the intended behavior, value, copy, ordering, or layout → **adopt it and bake the decision into the Fix plan.** Do NOT ask the dev a question the SOT already answers — cite the source in the plan instead (e.g. *"theo spec `book_chap_coin`=5"*).
2. **Then fall back to the dev's input.** If the digest hasn't landed, use the refs the dev supplied up front the same way — they are valid ground truth until memory supersedes them.
3. **Only THEN ask.** A `Câu hỏi cần làm rõ` is justified **only** when, after consulting the SOT and the dev's input, the answer is **genuinely absent** from both (no spec value, no figma node, the relevant external source is still `pending`), **or** the SOT/code contradicts itself / the logic is genuinely weird (a value that can't be right, two sources disagreeing, a fix that would violate an invariant). In those cases ask the one focused question — and say *why* (which source is silent / what contradicts what).
4. **Never ask about a `pending` external source as if the dev must answer it inline** — note it's awaiting a readable source (the capture re-tries each run, or the dev pastes it), implement what the accessible SOT pins down, and flag the dependent part. Don't manufacture a decision question out of a data gap that the SOT will fill.

> Litmus test before emitting a question: *"Is this answer findable in spec.md / figma.md / the external `<slug>.md` / the dev's provided refs?"* If yes → decide, cite, don't ask. If no, or the sources conflict / the logic feels wrong → ask, and name the gap. The default is **decide**, not ask.

## Multi mode — analyze fan-out
(The full turn loop — pull → pick → claim → fix → per-ticket review & batch-merge — lives in `references/phase4-fix-and-build.md`. This section owns only the **analyze** steps of that loop.)

After **Claim all** `[CLAIM]` (phase4 turn-loop step 4) and writing `.jira-bug/batch-progress.md`:

> **Stream results sequentially — never wait for all analyzers.** Analyzers run **concurrently in the background**, but the dev sees their plans **one at a time, in a FIFO-by-completion queue** with a **single interactive floor**. Critical mechanism — get this right or the dev waits for the slowest ticket:

1. **Fan out analyzers — background dispatch** (read-only, one per ticket — Multi fan-out block in `references/run-blocks.md`). **Launch EVERY picked ticket's analyzer as a background agent (`run_in_background: true`), all in ONE message.** This is the load-bearing detail: foreground-parallel tool calls return only after the *last* one finishes (that is the "waited for all 4" bug); background agents each fire a completion notification **independently**, so the first to finish can be presented while the rest keep running. Mark all `analyzing`. Each analyzer is handed the **`record_bug_status.md` ledger** (so it won't re-fix a root cause another ticket owns) + the resolved SOT context (Context-source rule above) + the Decide-from-SOT rule.
2. **Completion queue (FIFO by completion).** As each analyzer's completion notification arrives, **append that ticket to a ready-to-present queue in arrival order** (first analyzer done → head of queue, matching "196 xong → hiện"). Do NOT order by pick order — order by who finished first.
3. **Single interactive floor — present the queue head, one at a time.** Only ONE ticket holds the interactive floor. Take the head of the completion queue and: **FIRST, `memory-keeper` writes its memory to the shared remote immediately — upsert the ledger row in `project/<PROJ>/<PHASE>/session/record_bug_status.md` (`status: analyzed`, `root_cause_slug` + the concise root cause in `summary` + touched `file:line`s in `files`)** so the next analyzer/dedup — including other devs' sessions — sees the slug and the root cause. **This ledger row is the ONLY memory record per bug — there is no per-ticket file.** **The write is not done until it is pushed to `origin` (local-only = invisible to other devs = duplicated work); re-push the row on every later status change.** Then present by status (below). **While the floor is held, every other completion just enqueues (FIFO) — it must NOT interrupt the current dialog.**
4. **Atomic disposition → release the floor (fixer runs in the background).** Each ticket holds the floor for exactly ONE interactive exchange, then releases so the next queued head is presented immediately:
   - `plan-ready` → present its **FULL Fix-6 block VERBATIM** (all nine fields in order, never trimmed/summarized — see the MANDATORY rule under the Fix-6 block; every file ref = short-display markdown link `[File.kt:line](full/repo-relative/path:line)`, full path in the target) `[VN]`, ask `[OPTIONS]` (`Approve`/`Needs changes`). **`[AUTO]`: no ask — apply `references/helpers/auto-plan-pick.md` to the scorecard (≥80 → auto-Approve below; <80 / clarify-gap → mark `deferred` + report, no fixer, release the floor).**
     - **Approve** → mark `approved` (ledger) → **launch its fixer as a BACKGROUND agent now** (phase4 turn-loop step 8; sonnet, own worktree — runs in parallel with still-running analyzers and other fixers) → **release the floor**.
     - **Needs changes** → take the dev's direction → **re-dispatch a revised analyzer in the background (`run_in_background: true`)** → **release the floor IMMEDIATELY**; the revised result **re-enters the completion queue (FIFO) when it lands** (the dev does NOT block on the revise — they move to the next ready ticket and come back to this one when it re-surfaces). **HARD RULE — never do the re-analysis inline on the main loop.** This holds for EVERY kind of "Needs changes": a plan tweak, a deeper code dig, AND especially **"go analyze the media you skipped" (a video/image the first pass couldn't ingest)** — re-dispatch a background analyzer handed the dev's direction + the media paths + the Fix-5 fetch/frame-extraction recipe, then release the floor and present the next queued head. Running the revise (downloading the video, extracting frames, re-grepping) on the main loop is the bug this rule forbids — it freezes the dev on the slowest ticket. The only thing the main loop does on "Needs changes" is dispatch + release.
   - `duplicate` (same `root_cause_slug` as an in-flight/done ticket the analyzer saw) → tell the dev which ticket owns it, ask `[OPTIONS]` (`Skip`/`Fix anyway`) → resolve → release floor. Skip → mark `commented`/closed, no fixer.
   - `comment-needed` (→ Phase 4 Fix-9) → post the drafted `[VN]` comment `[REST]`, tell the dev the tech detail, ask disposition `[OPTIONS]`, mark `commented` → release floor.
   - `blocked` → surface, ask how to proceed, mark `blocked` → release floor.
5. **Deferral & drain.** After releasing the floor, present the next queued head at once (if the queue is empty, idle until the next completion notification arrives). **Loop until: the completion queue is empty AND every analyzer — including every `Needs changes` revise — has reported AND every disposition is resolved.** Then proceed to the windowed PR rounds (phase4 turn-loop step 9). Use the background-task completion notifications / `TaskList` to know which analyzers have finished while you were on the floor.

**Boundaries:** analyzers and fixers are background agents that **never touch the interactive floor** — only the main session presents/asks. Analyzers are read-only (no worktree/edits/Jira); approved plans cross into Phase 4 for the actual fix.

## Single-ticket workflow — analyze steps (Fix-1 … Fix-8)

**Fix-1 — Claim** `[CLAIM]` the first action (before fetch/analyze). Fail → stop, don't fetch. (In-Progress lock block, `references/run-blocks.md`.) **`[SHEET]`: no Jira transition — claim = upsert the `record_bug_status.md` ledger row `status: analyzing` keyed `<PROJECT>-SHEET-<N°>` (`references/google-sheet-board.md` → "Claim").**

**Fix-2 — Source-of-truth (background, non-blocking).** Phase 2 already launched the capture. Do **not** WAIT for it. Ensure the background `source-of-truth` capture is running (launch it now if Phase 2 hasn't, then proceed without waiting). Resolve SOT context for this ticket via the **Context-source rule** above (memory if ready, else the dev's provided refs). The run only stops if the **mandatory spec/figma links were never provided** in Phase 2 — in that case go back and ask (the ask is the only blocking part). (**Phase 2** — `references/phase2-source-of-truth.md`.)

> **`[TOKEN]` Run Fix-3 → Fix-6 in ONE dispatched analyzer subagent — NOT inline on the main loop.** Fix-4/Fix-5 download Jira attachments and Read **images and (especially) video**, and **Fix-5.5 reproduces on-device** (screencaps / screen-recording observed via human-mcp) — all heavy artifacts that must stay in the subagent's context, never the main loop's. Dispatch a single analyzer (Task tool; **read-only to the CODE** — `git grep`/`git show` only, never checkout/edit — but it **DOES drive the device for the Fix-5.5 repro**, acquiring the device lock; same agent multi mode fans out, just one here) handed the ticket key + SOT context + the `record_bug_status.md` ledger + the Fix-6 output format + `references/helpers/bug-analysis-rules.md`; it fetches, analyzes the media, **reproduces the bug**, and **returns ONLY the Fix-6 block text + the saved repro-evidence paths** (root cause + plan + files + scorecard). The main loop presents that block and runs the Fix-8 approval gate — it **never Reads the attachment images/video itself**. (Single mode is still "no fan-out" — this is one subagent for token isolation, not parallel analysis. Same `[TOKEN]` principle as the Phase-5 verifier.) The cheap main-loop steps stay inline: Fix-1 claim, Fix-2 SOT launch, Fix-7 clarify, Fix-8 approval.

**Fix-3 — Fetch ticket:** `mcp__jira__jira_get_issue(issue_key=KEY, fields="*all", comment_limit=50)`. Extract summary/description/status/priority/assignee/reporter/labels/components/comments. `attachment` often absent even with `*all` → re-fetch `fields="attachment,comment,description,summary"` or REST (Fix-4). **`[SHEET]`: there is NO Jira fetch — the analyzer already has the bug's `summary`/`details`/`expected`/`feature` + `attachments` URLs from the cached `setup.sheet` row; skip Fix-3/Fix-4 and go straight to media on those URLs (`references/google-sheet-board.md` → "Analyze & media": streamable→api, ibb.co→`og:image`, then human-mcp).**

**Fix-4 — Fetch attachments:** MCP can't download → REST (`references/helpers/jira-rest-api.md`). REST unavailable → ask the dev to paste images/logs.
> **`[TOKEN]` Downscale + recompress image attachments BEFORE vision** per `references/helpers/media-preprocessing.md` (≤1280px long edge, JPEG/WebP q80; `imagemagick`/`jimp`) — Read the shrunk copy, keep the original for evidence. Analysis-input only; Phase-5 decisive device captures stay full-res.

**Fix-5 — Analyze media:** **`[MEDIA]` route through human-mcp first (`references/helpers/media-analysis.md`)** — `mcp__human-mcp__eyes_analyze({ source: "<attachment path>", focus: "<the bug / UI state / repro step>" })` returns a TEXT understanding of the image OR video (pixels/frames never enter context). Use its text to extract what's wrong, expected vs actual (UI state, errors, device info; for video: repro steps, transitions, crash moment). **Only if human-mcp is exhausted-today / unconfigured** (availability cache in `media-analysis.md`) → fall back to Claude-native vision on the **downscaled copy from Fix-4** (`media-preprocessing.md`).
> **Video is NOT optional to analyze — exhaust the fetch+extract fallbacks before ever "ask the dev to summarize".** A video attachment / link is usually the clearest repro; skipping it (e.g. "MCP không tải được") produces a half-blind plan the dev will reject. Order of attempts:
> 1. **Get the real media file.** Jira attachment → REST download (`references/helpers/jira-rest-api.md`). **Hosted link** (Streamable / Loom / Drive / Imgur / direct URL) → resolve the underlying file: Streamable → `curl https://api.streamable.com/videos/<id>` then fetch `files.mp4.url`; Drive → the `uc?export=download` form; otherwise the direct media URL. Save under the scratchpad.
> 2. **Ingest it — `[MEDIA]` human-mcp first (`references/helpers/media-analysis.md`).** Pass the video file path straight to `mcp__human-mcp__eyes_analyze({ source: "<vid>", focus: "repro steps + the moment it breaks" })` → text back, no frame extraction, no bytes in context. **Only on the native-vision FALLBACK** (human-mcp exhausted-today / unconfigured) → extract frames with ffmpeg and Read them (Claude image vision): `fps=1` (NOT source 30fps), `scale=300:-1`, trim to the window `-ss <t> -t <dur>` — `ffmpeg -y -ss <t> -t <dur> -i <vid> -vf "fps=1,scale=300:-1,tile=<cols>x<rows>" montage.png`, Read the ONE montage; scene-change `select='gt(scene,0.3)'` for busy clips; higher-fps zoom only if 1fps missed the break. **Caveat: no ffmpeg `drawtext` unless fontconfig is set up — it errors on this stack.**
> 3. **Only if all of the above fail** (file unreachable AND no ffmpeg/vision path) → ask the dev to summarize (key steps + the moment it breaks), and say which fetch attempts failed.
> **Doing this re-analysis is a BACKGROUND agent, never inline** — see the `Needs changes` rule (multi) / Fix-8 (single): when the dev sends it back to look at media, re-dispatch the analyzer with the media paths + this extraction recipe as a background task and release the floor; do not run frame extraction on the main loop.

**Fix-5.5 — `[REPRO]` Reality-check + reproduce BEFORE root-causing (full rules: `references/helpers/bug-analysis-rules.md`, `A1`–`A14`).** This runs inside the analyzer subagent, before Fix-6.
1. **Reality check (A1–A3):** classify real code bug vs a non-code cause (Fix-9 set — config/console · backend · stale build/env/device · works-as-designed · duplicate · cannot-reproduce), citing evidence; non-bug → route to Fix-9 / defer, don't invent a fix. Ledger dup-check (`root_cause_slug`) first.
2. **Reproduce on-device (A4–A6):** acquire the device lock (`references/helpers/device-lock.md` — serialized with Phase 5, bounded wait; owner token = the RAW session id, every adb call through `device-lock.sh exec`, `BUSY` → wait/another serial, never bare `adb -s`), follow the ticket+media repro steps, **observe via `[MEDIA]` human-mcp** (`eyes_analyze`). **Save the repro evidence** (image=static, video=dynamic/crash) to `.jira-bug/evidence/<TICKET>/` — Phase 5 reuses it as the "before" state. Then **trace the trigger→symptom code path** to a `file:line` on `origin/<BASE>`.
3. **Device policy (A7):** **`--auto` → device HARD-REQUIRED, no static fallback** — no device / lock unobtainable in the bounded wait / repro fails → **defer + report** (`--auto` never auto-fixes an unreproduced bug). **Interactive → repro-first, degrade gracefully** — no device/lock busy → static evidence-based repro + **flag `not device-verified`** in the Fix-6 block; the dev decides.
4. **Can't-reproduce ladder (A8–A10):** repro steps don't trigger it → drive candidate cases (empty/first-run/logged-out/offline state · remote-flag · timing/race · data/locale variant · device-OS · build variant); one triggers → that's the condition (note it); none after the ladder → **defer as unreproducible, NEVER guess-fix** (interactive: comment the reporter for exact steps/video/build + move to `Request`; `--auto`: defer + report).
5. The confirmed repro steps **become the Phase-5 verify script** (A12); the repro result keys the `deterministic_repro` scorecard points (A4/A9 → full, A10 → 0). **Multi/team:** repro is **serialized best-effort** on the one lock (timeout + retry cap) — lock-miss under `--auto` → defer that ticket, under interactive → static + flag.

**Fix-6 — Synthesize root cause + fix plan** (per bug). **Always analyze against the NEWEST base code — `git fetch origin` FIRST, then search/grep/read the code as it exists on the latest `origin/<BASE>`** (the project's integration branch = `setup.json.baseBranch`, the SAME ref Phase 4 will branch the fix from), **NOT** the local working tree (which may be stale, dirty, or on another branch). Read files at that ref directly — `git grep <pat> origin/<BASE>` / `git show origin/<BASE>:<path>` — so every `file:line` you cite is accurate against `origin/<BASE>`. Analyzers stay **read-only**: fetch + read refs only, never check out / create a worktree / edit (that is Phase 4). If `<BASE>` is somehow still unresolved here, resolve it via the **Base branch** ASK gate (`references/run-blocks.md`) before grounding the analysis. Investigate **silently** (code search/grep/blame/hypotheses internal). User-facing output is ONLY the block below — never stream the raw log or competing-hypothesis hedges. **Converge on ONE root cause**; if evidence is split, pick the single most-likely and put the one confirming check in *Câu hỏi cần làm rõ* (don't present (a)/(b)/(c) as the answer). `[VN]` plain wording, tokens verbatim. **The `Mô tả`, `Comment`, and `Resource link` fields are pulled straight from the fetched ticket (Fix-3) + attachments (Fix-4)** so the dev can click through to the original source. > ```

This is both short on screen AND clickable-to-exact-line.

> - **Inline refs** (inside `Root cause` / `Fix plan`) — same short-display markdown link inline (e.g. `[SeriesMapper.kt:22](app/src/.../SeriesMapper.kt:22)`); concise on screen, still clickable. Don't print the long bare path on screen — it lives inside the link target.

# Init phase — Multi mode (solo worker, no team)

> Loaded by SKILL.md's mode dispatch when the invocation resolves to **multi** with NO `--team` (`--mode multi`, the `pull`/`batch` aliases, or the default when no mode/ticket is given). This is the **solo worker**: pull the board → list bugs by category → dev picks → fix the set in fan-out + windowed PR rounds → re-pull next turn. Self-contained: the **Pull & list by category** block lives HERE (the canonical copy — the phase4 re-pull and the team file both point back to it). When you reach the workflow read `references/phase4-fix-and-build.md` (Multi mode turn loop; analyze fan-out is in `references/phase3-analyze.md`); the **Source-of-truth capture** is Phase 2 (`references/phase2-source-of-truth.md` — background, non-blocking), other shared gates live in `references/run-blocks.md`. Cross-cutting invariants are SKILL.md **Golden Rules** — cited by tag, not restated.

## Init-1…5 — one script, then act on the result

> **No project key in the invocation? ASK FIRST — before running init.** If the user ran a bare `/jira-bug-analyzer` (no key arg, no `--team`), your FIRST action is an `AskUserQuestion` `[OPTIONS]` for the project key. **Do NOT infer the board from the working-directory name, the repo, or `CLAUDE.md`, and do NOT silently adopt a cached/remote board.** If a local `.jira-bug/setup.json` exists, you MAY read its `board` and offer it as the **recommended first option** (one-tap confirm) alongside an "Other" free-text fallback — but the user must still choose. Only once you have a key do you run `init-multi.sh --project <KEY>`. When the key WAS given as an arg, skip this — use it verbatim (no prompt).

Run the multi-mode init probe once at start, passing the **target bug repo root** and, **when the project key is known, `--project <KEY>`** so init can resolve setup as **local cache → remote memory → absent**:

```bash
bash <skill-dir>/assets/init-multi.sh "<PROJECT_ROOT>" --project <KEY>   # --project lets init reuse a teammate's project/<KEY>/setup.json when there is no local cache; add --recheck-env to force a fresh env probe
```

**Pass `--project <KEY>` — the project key the skill was invoked with** (`/jira-bug-analyzer AIP686` → `AIP686`, the SKILL.md dispatch rule 4, or an explicit `--board`). **The provided key is trusted verbatim as the board — do NOT validate it against `get_all_projects`.** Only when NO key was provided (bare default-multi) ask once for it (Init-9 board step), then **re-run `init-multi.sh --project <KEY>`** (cheap — env + memory are cached) so init can do the remote-setup check before treating setup as `absent`.

It prints an `INIT-STATUS` block (`MODE=multi`, probes **only the `TIER=now` deps** via `preflight-env.sh --now-only` unless a fresh env cache exists, `memory-sync.sh pull 600`, reads `.jira-bug/*` + resolves setup local→remote — all read-only, cached, never blocks, exits 0). On a warm cache it also emits **`PULLQUERY_JQL=<jql>`** so you fire `jira_search` straight from the script output — **do NOT `Read` `setup.json` separately**. Both the env-probe result AND the `MCP_SETUP` status are cached together in the **user-scope** `~/.claude/jira-bug-analyzer/probe-cache.json` (30d TTL, machine-local, never pushed, shared across all projects on this machine — both derive from `~/.claude.json`, not the repo; a legacy project-level `.jira-bug/env-cache.json` is auto-migrated on first run). `mcp_setup` is stored as a **per-server map** (`{"jira":"ok","confluence":"ok","figma":"ok","human-mcp":"ok"}`). On a warm cache init skips **both** the preflight and the `setup-mcp.sh status` fork; `--recheck-env` bypasses it. The cache is treated as an MCP hit only when **every** template server is `ok`, so a missing server (or a newly-added template server) re-probes on the next run and self-heals the moment setup completes. `later`/`cond` deps (gh/adb/gradle/figma/confluence/mobile/skills) are NOT probed here — they're re-checked just-in-time at their gates. **In the same first turn also fire `ToolSearch select:mcp__jira__jira_search`** to preload the pull tool. **First-run persist (bootstrap exception — the ONE write allowed at init). Persisting config is TWO writes, not one — local mirror AND shared remote — and the shared push is MANDATORY, never optional, never deferred** (same hard rule as the `baseBranch` gate). Shared `project/<PROJ>/setup.json` carries the resolved `pullQuery` **and `baseBranch`**, known only after the first pull / base resolution. So whenever config is missing (`SETUP=absent` → intake): intake → pull → **the moment the pull succeeds, persist BOTH layers in the SAME turn**:
1. **Local mirror:** write `.jira-bug/setup.json` **via `bash <skill-dir>/assets/setup-json.sh merge-set` — NEVER hand-write the JSON** (a hand-written trailing comma reads back as empty under strict parse and silently breaks config reuse).
2. **Shared remote (cross-dev) — MUST happen the same turn:** write `project/<PROJ>/setup.json` with the **same canonical writer** (`setup-json.sh merge-set`, stamp `savedBy`/`savedAt`) — via the `memory-keeper` agent (`action: project-setup`) — then push. A local-only persist is a **BUG** — the next dev/teammate won't inherit the config and gets re-asked. "Asked the dev but didn't reach remote memory" is exactly the failure this rule forbids: asking is not persisting; the config isn't saved until the shared `setup.json` is on `origin`.

Do this *before/alongside* the list, **not** deferred. If the push fails, surface it — do not silently leave config local-only. Every OTHER run write (ticket claim/status, secrets, setup *changes*, KB/session pushes) is deferred to ticket-pick. The `env` block is machine-specific — **local only, NEVER pushed**.

**Resolve the base branch at init (persist for reuse).** Whenever `baseBranch` is absent (`BASEBRANCH=none`), **ASK the dev — this is a MANDATORY interactive gate, never a silent resolve.** Present an `AskUserQuestion` whose options are the resolved candidates (recommended first + `Other` free-text) and store the dev's pick as `setup.json.baseBranch` (the **Base branch** block in `references/run-blocks.md` has the full ASK rule + candidate order: `BASEBRANCH_SUGGEST` → CLAUDE.md/contributing docs → remote default branch → checked-out integration branch; CLAUDE.md wins on conflict, the dev's pick always wins). A confidently-resolvable value is only the prompt's default — it does NOT license skipping the ask. Every worktree branches from `origin/<baseBranch>` and every PR targets it (`--base <baseBranch>`) — **never assume `main`**. On later runs, `baseBranch` is read from the cached/shared setup like `pullQuery` and surfaced in INIT-STATUS as `BASEBRANCH=<branch>` — present → use it, no prompt. **When it's missing, init NEVER auto-fills it — the dev is ASKED** (`BASEBRANCH=none` forces `read-phase1`; see the `BASEBRANCH=none` row above). init only *reads* the shared `project/<PROJ>/setup.json` value (when present) to emit `BASEBRANCH_SUGGEST=<v>` — a **recommended one-tap default**, never a silent write. A removed/absent `baseBranch` is a signal to (re)configure, so the dev confirms (or changes) the value; their pick is then persisted local + shared. **Exception:** a wholesale `SETUP=remote` adoption (no local mirror at all → adopt a teammate's complete setup) brings `baseBranch` along with it silently, like every other field. Pushing the chosen value to shared `project/<PROJ>/setup.json` lets other devs inherit the same base. **Do NOT branch from `origin/main` when `baseBranch` differs** — that pollutes the PR diff with the main↔base divergence and forces a late rebase.

**Setup resolution nuance.** init resolves setup deterministically as **local → remote → absent**. When local exists but a *newer/differing* shared `project/<PROJ>/setup.json` was pulled, **adopt the shared one** (refresh local mirror, one-line note naming `savedBy`). `--reconfigure`/args override → re-run intake for the changed fields and **re-persist BOTH layers — local mirror AND a shared push via `memory-keeper` whenever a project-level field changed** (board/status/assignee/spec/figma/otherRefs/pullQuery/baseBranch). The shared push is **mandatory the same turn**, exactly like first-run — a changed field that lives only in the local mirror leaves teammates on stale config. Never re-push an unchanged setup, never push `env`. **Args always win.**

Force a fresh `pull 0` only on `--resume` (the post-analysis dedup reads the Init `pull 600` copy — no extra sync at pick/claim time). **The memory repo is FIXED — `https://github.com/hung-apero/jira-bug-memory` — no fallback, no ask;** clone absent → `memory-sync.sh autowire` (clones it), `REPO_ACCESS` → surface the access error (gh auth + repo permission), never local-only. **MCP setup runs HERE at init (template-driven, AUTO):** `setup-mcp.sh status` reads the template and reports `MISSING=`/`NEEDS_SETUP=`; `NEEDS_SETUP=yes` → **auto-run `setup-mcp.sh all --token <tok>` immediately (no "shall I install?" prompt)** — it provisions jira+confluence+figma+human-mcp at once (skips any present / live-via-other-config). ### Init-6 — Model (per-stage policy — do NOT ask)
**The skill's default session model is `sonnet`.** Each stage then runs on the model that fits its cost/difficulty. **Do not ask for a model up front** — apply this matrix automatically:

| Stage | Model | Why |
|---|---|---|
| Get Jira ticket (board pull, fetch, attachments, search) | **haiku** | cheap I/O + listing |
| Board bug-list categorization (semantic grouping of the pull for the pick list) | **inline on main (default `sonnet`)** | runs directly in the main loop — never a subagent; the default `sonnet` main loop groups a short list fast, and a spawn round-trip would only add the lag the dev sees |
| Analyze + fix plan (root cause, scorecard, plan) — the analyzer | **opus** | hardest reasoning, deep root-cause |
| Fix the bug (implement per approved plan) — the fixer | **sonnet** | solid code execution |
| Categorize into a KB category + dedup match — memory-keeper's judgment work | **opus** | hardest judgment — wrong category pollutes KB, missed dedup wastes a fix |
| Everything else (board pull/fetch/attachments, Jira writes, comments, worklog, PR finalize, plain status-file writes) | **haiku** | cheap mechanical steps |

- **Spawn each subagent on its stage model** via the Agent tool's `model` param: analyzer → `opus`, fixer → `sonnet`, memory-keeper / fetch helpers → `haiku`. The board bug-list categorization is **NOT** a subagent — it runs **inline on the main loop** (default `sonnet`); see the listing block.
- **`--model <opus|sonnet|haiku>` overrides the whole matrix** — forces that one model for every stage (escape hatch). Persist it to setup memory; without it, the per-stage matrix applies (no model prompt).

### Build the query
JQL scoped to board + status set + assignee scope (Init-9), active sprint, always excluding flagged:
```
project = "<BOARD>" <ASSIGNEE_FILTER> AND issuetype = Bug AND <STATUS_FILTER> AND sprint in openSprints() AND Flagged is EMPTY ORDER BY priority DESC
```
- `<STATUS_FILTER>` (default `Request`+`Reopened`): one → `status = "<S>"`; many → `status in ("<S1>","<S2>")`.
- `<ASSIGNEE_FILTER>` (default Any → omit clause): me → `AND assignee = currentUser()`; Unassigned → `AND assignee is EMPTY`; user → `AND assignee = "<u>"`.
- **`Flagged is EMPTY` always present** — a flagged (impediment) ticket is never listed or resolved.
- **Active-sprint default; Kanban fallback:** if the query errors on `sprint` or is empty *because the board has no sprints*, drop the clause, re-run once, tell the user it fell back to all bugs. Never return empty solely because the board is sprintless.
- **Board key is used verbatim — NO validation.** The dev provided the project key (invocation arg / `--project`); feed it straight into the JQL. No `get_all_projects` lookup, no fuzzy-stem check, no post-query prefix sanity. If the key is wrong the pull simply returns empty (or a JQL error) → tell the user and let them re-supply it; never silently re-resolve to another project.
- **Poll-speed cache:** on success, persist `pullQuery = {jql, sprintMode:"sprinted"|"sprintless", savedAt}` in `.jira-bug/setup.json`. Subsequent pull with unchanged board+status+assignee (re-pull turn / `--resume` reconcile) → fire the cached `jql` in ONE query (skip the sprint-clause probe). ### List the bugs grouped by category
- **Output: grouped by semantic Category — infer each bug's feature / functional area from the FULL summary text (LLM judgment), NOT a bracket parse.** Read what the bug is actually about and assign a concise feature-area label. The leading `[...]` tag is only a **hint**, not the group key: split a coarse tag by meaning (e.g. `[Book]` bugs about reading-flow vs. chapter-unlock vs. detail-UI belong in *different* categories) and merge differently-tagged bugs that share one feature area (e.g. an `[Audio book / Book]` unlock-coin bug groups with a `[Coin]` bug). Keep labels short, Title Case, original-language tokens preserved (e.g. `Dịch`). No clear feature area → `### Uncategorized`. This spends a little upfront reasoning to produce **meaningful** buckets the dev can pick from, instead of coarse tag lumps. Example labels: `Reading Experience` · `Chapter Unlock & Coin` · `Book Detail UI` · `Home Layout` · `Comments` · `Localization` · `Rating`.
- **Run this categorization INLINE on the main loop — NEVER delegate it to a subagent.** The skill's default model is `sonnet` (see Init-6), which is fast enough to group a short bug list inline, so the grouping always happens directly in the main loop the instant `jira_search` returns — no `Agent`/subagent spawn, no `run_in_background`. The spawn round-trip would only add latency the dev sees. This applies regardless of the session model (even if `--model opus` was forced — still inline; the dev opted into the slower model). It does NOT change the grouping rules above, and it is the board **pick list** only, distinct from the `opus` KB-categorize+dedup row in the Init-6 matrix.
